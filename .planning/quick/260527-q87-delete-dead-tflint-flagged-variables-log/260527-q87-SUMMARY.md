## Why these are safe deletes (traceability)

1. **`logout_redirect_uri`** (`infra/external/variables.tf:18-22`): Phase 4 D-12 logout flow is functionally complete via the SPA's `redirect_uris` list — UAT passed in `.planning/phases/04-frontend-shell-auth/04-06-SUMMARY.md:176` ("Sign-out: ✅ PASS"). This variable was scaffolded in Plan 04-01 against Pitfall 11 (Entra returning 400 on unregistered post-logout URI) but was never threaded into any `azuread_application_*` resource in `infra/external/main.tf`. Entra accepts post-logout redirects to URIs already in the SPA `redirect_uris` list — verified by working production logout.

2. **`swa_origin`** (`infra/modules/identity/variables.tf:1-5`): The identity module's own `main.tf` never read `var.swa_origin`. The real DEPL-12 two-pass for SWA-origin → SPA redirect_uris flows through `infra/external/` via `spa_redirect_uris`, NOT through the identity module. The identity module's pass-through was misrouted plumbing — a parameter that goes nowhere.

## Negative-space contract held

The env-level `variable "swa_origin"` in `infra/envs/{prod,dev}/variables.tf` was deliberately NOT touched — it is still consumed by:

- `infra/envs/prod/locals.tf:11,15` — `allowed_origins_csv` for ACA CORS allowlist
- `infra/envs/dev/locals.tf:14` — same CORS path on dev
- `infra/external/` invocation — env composition still passes the SWA origin to the external module via `spa_redirect_uris`
