## Implementation Decisions

### A. Profile DB read-path flip (PROF-01)

- **D-01: `load_profile()` body-flips to async DB query.** Replace the `data/profile.json` read with `SELECT * FROM user_profile WHERE user_id = :user_id`. Phase 1 D-07 explicitly built the kwarg-only signature (`*, user_id: UUID | None = None, path: str | None = None`) as a forward hook for this exact moment. Phase 5 D-06 deliberately deferred this flip to Phase 7. Counterfactual considered: leave the JSON file plus a "fallback to DB if file missing" path — rejected, ambiguous source of truth conflicts with PROF-01 literal ("`data/profile.json` is no longer the read path").

- **D-02: `load_profile()` becomes async with an `AsyncSession` parameter.** New signature: `async def load_profile(session: AsyncSession, *, user_id: UUID | None = None) -> UserSkillProfile`. The `path` kwarg is removed (was Phase 1 D-07 forward-compat; v1 callers pass no `path` so removing it has no production-call-site break). All API call sites already run in async handlers with `Depends(get_session)`; CLI agent uses the sync engine through a Sessionmaker-bound short-lived AsyncSession (acceptable) or — cleaner — the CLI `agent` command runs through `asyncio.run` against an `AsyncSessionLocal`. Counterfactual: dual sync+async overload via `functools.singledispatch` — rejected, two code paths means double the bug surface for one user.

- **D-03: Seed Adrian's profile via Alembic data-migration `00NN_seed_user_profile.py`.** Embed `data/profile.json` content as a Python dict literal inside the migration (NOT a file read at runtime — the migration must be runnable against a fresh image where `data/` is gitignored at the container layer). Upgrade: idempotent UPSERT (`INSERT … ON CONFLICT (user_id) DO NOTHING`). Downgrade: `DELETE FROM user_profile WHERE user_id = :seeded_uuid` for symmetry. The migration depends on whichever migration introduced `user_profile` (Phase 1 plan 01-02 baseline) and Phase 4 `0005_adopt_entra_oid`. Counterfactual: `init_db()` idempotent seed — rejected, `init_db` is for boot-time invariants (Phase 4 D-10 entra_oid UPDATE pattern); a one-time profile backfill is canonically a migration.

- **D-04: `data/profile.json` repurposed as test fixture; not deleted.** File stays in `data/` as a reference snapshot. Backend tests + the Alembic seed migration reference the same JSON content (the migration embeds the dict literal generated from this file). Update `data/README.md` (NEW small doc, 10 lines) to record the file's new role: "reference snapshot, NOT a runtime read path — see PROF-01". Counterfactual: delete the file — rejected, useful as a human-readable record of v1 seed contents and helpful for future reconstitution if the DB seed migration is misapplied.

- **D-05: `data/profile.json` deletion-as-canonical-path is verified in two ways at PR time.**
  1. `grep -rn "profile.json" src/` returns ONLY the data/README.md doc reference (no production read paths)
  2. A backend test patches `settings.profile_path` to a nonexistent path and asserts `load_profile()` still works (proves the file is not read at runtime)

### B. Upload endpoint contract (PROF-02)

- **D-06: Endpoint = `POST /profile/upload`** with `multipart/form-data` body, single `file: UploadFile` field. Returns `ResumeUploadResponse` (D-13). Wired with `dependencies=[Depends(require_api_key), Depends(standard_limit)]` (30/min, same tier as `/search` / dashboard endpoints) + `user_id: UUID = Depends(get_current_user_id)`. AUTH-06 single-user guard fires uniformly per Phase 4 D-08. Counterfactual: dedicated `upload_limit = 5/min` — rejected, hand-tuning rate-limit tiers per endpoint inflates the surface; standard_limit suffices for v1.

- **D-07: 2 MB cap enforced via `Content-Length` header BEFORE body read** (REQ PROF-02 literal: "rejected with a 413 before the body is fully read"). Implementation: a tiny ASGI-level check OR a route-level dependency that inspects `request.headers.get('content-length')` and `raise HTTPException(413, …)` before `UploadFile` materializes. Fallback for chunked-encoding (no Content-Length): wrap `await file.read()` in a size-accumulating loop that aborts at 2 MB. Constant: `MAX_RESUME_SIZE_BYTES = 2_000_000` in `config.py` Settings field (overridable via env). Counterfactual: rely on Starlette's silent body-stream cap — rejected, Starlette has no first-class size cap, and rejecting AFTER the body read violates the literal REQ wording.

- **D-08: File-type whitelist = extension + Content-Type intersection.** Accept iff `Path(filename).suffix.lower() in {".pdf", ".docx"}` AND `Content-Type in {"application/pdf", "application/vnd.openxmlformats-officedocument.wordprocessingml.document"}`. 415 with `{reason: "unsupported_file_type"}` otherwise. No magic-byte sniffing (python-magic adds a libmagic native dep — overkill for two file types and a single user). Counterfactual: extension-only — rejected, malicious .pdf-renamed-EXE bypass; extension+MIME is the standard browser-shipped contract.

- **D-09: PDF parser = pypdf 6.x (REQ-PROF-02 literal); DOCX parser = python-docx 1.x (REQ literal).** Both added to `[project] dependencies` in `pyproject.toml`. PDF: iterate pages, concatenate `page.extract_text()` joined by `\n\n`. Encrypted PDF → `pypdf.errors.PdfReadError` caught → 422 `{reason: "pdf_encrypted"}`. DOCX: iterate paragraphs + tables, paragraphs joined by `\n`, table rows by `\n`, cells by `\t`. Headers/footers ignored (paragraphs only — `doc.paragraphs`).

- **D-10: Text extraction post-condition: ≥100 non-whitespace chars.** If `len(text.strip()) < 100`, return 422 `{reason: "text_extraction_failed"}`. Catches scanned/image-only PDFs that "succeed" but yield empty text. Threshold rationale: any legitimate resume has ≥100 chars; <100 implies extraction noise.

- **D-11: Text size cap pre-LLM = 50_000 chars (~12 KB).** Truncate at 50 KB if larger; emit a `log.warning("resume_text_truncated", original_chars=N, truncated_chars=50_000)`. Protects against unbounded LLM cost on weird/huge inputs (a CV with 50 pages of code samples). Average AI-Engineer resume is 3-8 KB plain text; 50 KB is 6× the practical max.

### C. Extraction prompt + Instructor (PROF-03)

- **D-12: New module `src/job_rag/extraction/resume_prompt.py` with `RESUME_PROMPT_VERSION = "1.0"`.** Pinned constant; bump on prompt changes (Phase 2 D-22 pattern). Module exports `RESUME_SYSTEM_PROMPT` + `RESUME_PROMPT_VERSION`. The version travels in the upload response (D-13) and Langfuse trace metadata (D-29). Future evolution paths: explicit soft-skill ban list (importable from `extraction/prompt.py`), structured `years_experience` extraction tightening, multi-language support.

- **D-13: Resume extraction output schema = NEW `ResumeExtraction` Pydantic model** (not `UserSkillProfile` directly). Fields:
  ```python
  class ResumeExtraction(BaseModel):
      skills: list[UserSkill]
      target_roles: list[str] = []
      preferred_locations: list[str] = []
      min_salary_eur: int | None = None
      remote_preference: RemotePolicy = RemotePolicy.UNKNOWN
      years_experience: int | None = None
  ```
  Lives in `src/job_rag/models.py` alongside `UserSkillProfile`. Mapped to `UserSkillProfile` for diff computation by a simple field-copy function. Separating the schemas means the extraction format can evolve (e.g., add `companies_worked_at`) without coupling the canonical user profile shape. Counterfactual: reuse `UserSkillProfile` directly — rejected, conflates the LLM's output shape with the canonical user state.

- **D-14: Prompt rejects soft skills (consistency with Phase 2 D-22 pattern).** Reuse the `REJECTED_SOFT_SKILLS` tuple from `extraction/prompt.py` (`("communication", "teamwork", "problem-solving", "leadership", "analytical thinking")` — Phase 2 D-22). Document the rejection list in the resume system prompt with carve-outs for spoken languages ("English", "German", "Polish") which Adrian's profile contains. Counterfactual: extract everything, filter in diff — rejected, wastes LLM tokens and pollutes the diff with noise that the user has to manually untick.

- **D-15: Instructor model = GPT-4o-mini.** Same as `extract_posting()` in `extraction/extractor.py`. Wrapped in:
  ```python
  @retry(wait=wait_exponential(min=1, max=10), stop=stop_after_attempt(3))
  def extract_resume(text: str) -> ResumeExtraction:
      ...
      return client.chat.completions.create(
          model=settings.openai_model,
          response_model=ResumeExtraction,
          messages=[{"role": "system", "content": RESUME_SYSTEM_PROMPT},
                    {"role": "user", "content": text}],
      )
  ```
  Lives in `src/job_rag/extraction/resume_extractor.py` (mirrors `extraction/extractor.py` placement). Sync function (consistent with `extract_posting`); called from the async route via `await asyncio.to_thread(extract_resume, text)` to avoid blocking the event loop (Phase 1 D-05 reranker pattern).

- **D-16: After 3 failed Instructor retries → 422 `{reason: "extraction_failed"}`.** No fallback to empty-skill profile. The user explicitly initiated the upload; silently saving an empty profile would destroy their existing skills on save. Logged as `log.error("resume_extraction_failed", attempts=3, exc_info=True)`.

### D. Diff computation + response shape (PROF-04)

- **D-17: Diff is stateless — single round-trip envelope.** Upload returns the full diff in one response. No "draft" row, no upload session, no two-step protocol. Response shape:
  ```python
  class SkillDiffItem(BaseModel):
      name: str
      source: Literal["added", "removed", "unchanged"]
      editable: bool  # True only for "added" (renames before save)

  class ResumeUploadResponse(BaseModel):
      extracted: ResumeExtraction        # full extraction (for transparency)
      skills_diff: list[SkillDiffItem]   # flat list with status tags
      prompt_version: str
      extraction_id: UUID                # for Langfuse trace correlation (D-29)
  ```
  Client owns the review state in React component state. PATCH save (D-21) accepts the final confirmed skills list — the server has no record of what was extracted vs what was already there. Counterfactual: persist a `resume_upload_draft` table — rejected, adds schema + cleanup + race conditions for a single-user workflow that completes in ≤60 seconds.

- **D-18: Diff envelope uses a FLAT list with status tags (not three separate lists).** Easier for the UI to render in a single column (D-25) and preserves a stable ordering (alphabetical or extraction-order — see D-19). Counterfactual: `{added: [...], removed: [...], unchanged: [...]}` separate lists — rejected, UI then has to concat-and-sort anyway, and "Unchanged" being the majority would push "Added" off-screen if rendered separately.

- **D-19: Diff item ordering = added first, then removed, then unchanged, each section alphabetical.** Most-relevant-first (the user mostly cares about additions, then deletions, then unchanged). Within each section, alphabetical for predictability. Implemented in `compute_skills_diff()` not in the frontend.

- **D-20: "Unchanged" defined via `_normalize_skill()` from `matching.py`.** Two skills are "the same" iff `_normalize_skill(a) == _normalize_skill(b)`. Alias-aware via `_ALIAS_INDEX` once `_ALIAS_GROUPS` is populated (currently empty per Phase 1 D-12; Phase 5 D-Discretion deferred populating it; Phase 7 does NOT populate). If alias collisions surface as a real problem post-Phase-7 (e.g., "AWS" + "Amazon Web Services" both ending up "added"), a focused follow-up phase populates `_ALIAS_GROUPS`. Counterfactual: implement alias-aware diff inline in Phase 7 — rejected, scope creep into a cross-cutting concern.

### E. PATCH save semantics (PROF-06)

- **D-21: `PATCH /profile` endpoint with full replace semantics on `skills`.** Request body:
  ```python
  class UserProfileUpdate(BaseModel):
      skills: list[UserSkill]
      target_roles: list[str] | None = None
      preferred_locations: list[str] | None = None
      min_salary_eur: int | None = None
      remote_preference: RemotePolicy | None = None
      extraction_id: UUID | None = None  # for Langfuse trace correlation (D-29)
  ```
  - `skills` is REQUIRED and REPLACES the row's `skills_json` entirely (the UI computes "the final list" by combining ticked added + ticked unchanged + ticked removed).
  - Other fields are OPTIONAL — `None` = no change. UI passes the user's confirmed values back; v1 review panel only edits `skills` (D-25) so UI sends `None` for the others.
  - SQL: `UPDATE user_profile SET skills_json = :s, …, updated_at = now() WHERE user_id = :uid`. UPSERT semantics not needed (the seed migration D-03 guarantees the row exists).

- **D-22: Save returns the persisted `UserSkillProfile`** so the client can update its TanStack Query cache via `queryClient.setQueryData(['profile'], data)` post-mutation. Phase 5 dashboard's `cv-vs-market` widget reads profile via `load_profile()` server-side; the new skills propagate on the next `/dashboard/cv-vs-market` fetch (cache invalidation via `queryClient.invalidateQueries({queryKey: ['dashboard']})` after save).

- **D-23: No optimistic concurrency / version vector in v1** (single user, no concurrent writes). `updated_at` advances purely for audit; no `If-Match` ETag handling. Future multi-user adds optimistic concurrency.

### F. Review panel UX (PROF-05)

- **D-24: Single-column status-pill chip list (NOT side-by-side current-vs-extracted).** Reasons: (a) for ~80 skills, side-by-side wastes horizontal real estate and forces eye saccades; (b) status pill (`[+ ADDED]` / `[− REMOVED]` / `[= UNCHANGED]`) reads denser and scans faster; (c) matches Linear-dense aesthetic per Phase 4 D-20 / Phase 6 D-31. Status pill colors via shadcn `Badge` variants (Phase 4 primitive):
  - Added → green pill `bg-green-500/10 text-green-700 dark:text-green-400`
  - Removed → red pill `bg-red-500/10 text-red-700 dark:text-red-400`
  - Unchanged → muted pill `bg-muted text-muted-foreground`

- **D-25: Default tick states = added✓, removed☐, unchanged✓.** "Accept the extraction wholesale" is the path of least resistance — one click "Save" produces "replace my current profile with the extracted set". User unticks specific added skills they don't endorse and re-ticks specific removed skills they want to keep. The save button's summary label updates live: `"Save (3 new · 2 keep removed · 47 unchanged)"`. Counterfactual: all-off default — rejected, requires the user to tick every unchanged skill, which is annoying and error-prone.

- **D-26: Inline name editing via shadcn Input inside chip on Pencil-icon click.** Pencil affordance shown only on "added" chips (PROF-05 literal: "edit skill names before saving"). Click pencil → chip body replaces with `<Input>` pre-filled with current name; Enter saves the new name in component state; Esc cancels. Removed and unchanged chips are read-only — editing canonical names retroactively is a v2 feature.

- **D-27: Layout = single shadcn Card with sticky footer.**
  ```
  <Card>
    <CardHeader>
      <CardTitle>Review extracted skills</CardTitle>
      <CardDescription>23 skills found · 5 new · 2 removed · 16 unchanged</CardDescription>
    </CardHeader>
    <CardContent className="max-h-[60vh] overflow-y-auto">
      <ul>
        {diff.map(item => <SkillDiffChip key={item.name} {...item} />)}
      </ul>
    </CardContent>
    <CardFooter className="sticky bottom-0 bg-background border-t">
      <Button variant="outline" onClick={cancel}>Discard</Button>
      <Button onClick={save} disabled={isSaving}>
        Save profile (N skills)
      </Button>
    </CardFooter>
  </Card>
  ```
  Footer always reachable; content scrolls. Mobile: footer is fixed-position (full bleed).

- **D-28: Route = `/profile`** (existing PhasePlaceholder stub in `frontend/src/routes/Profile.tsx` per Phase 4 D-Discretion). Two-state via React state in `<ProfilePage>`:
  - `idle` — shows `<ProfileView>` (current skills as read-only chips) + `<ResumeUploader>` dropzone
  - `reviewing` — shows `<ReviewPanel>` after a successful upload response
  Single route, conditional render. Counterfactual: nested route `/profile/review` — rejected, no deep-link value and the diff data lives in component state (refreshing `/profile/review` would lose the upload).

- **D-29: Component tree mirrors Phase 5 D-15 / Phase 6 D-27 feature-folder pattern.**
  ```
  frontend/src/routes/Profile.tsx           — page: idle / reviewing state machine
  frontend/src/components/profile/
    ProfileView.tsx                          — current profile read-only display
    ResumeUploader.tsx                       — file input + drag-drop + size pre-check + submit
    ReviewPanel.tsx                          — Card with diff chip list + footer
    SkillDiffChip.tsx                        — one row: tick + status pill + name (editable on "added")
    useResumeUpload.ts                       — TanStack mutation hook; owns upload + post-extract review state
    types.ts                                 — DiffItemState (extends SkillDiffItem with `checked` and `editedName`)
  frontend/src/api/profile.ts                — fills the existing stub with uploadResume() + saveProfile()
  ```

- **D-30: `<ResumeUploader>` = shadcn `Input type="file"` + drag-drop hint.** No third-party `react-dropzone` library. Implementation:
  - `<input type="file" accept=".pdf,.docx" onChange={handleFile}>` styled as a dashed-border drop zone
  - `onDragOver` / `onDrop` listeners on the wrapper div for drag-drop; `e.dataTransfer.files[0]` then through the same handleFile path
  - Pre-check: reject `>2_000_000 bytes` client-side before POST (saves a round-trip; backend D-07 is the authoritative backstop)
  - Pre-check: reject extension not in `{".pdf", ".docx"}` client-side; same logic as backend D-08
  - Shows file name + size + a "Remove" affordance before user clicks "Upload"

- **D-31: Loading state during upload = stepped status copy.**
  - 0-2s: `"Reading your resume…"`
  - 2-10s: `"Asking the agent to extract skills…"`
  - 10s+: `"Still working — extraction can take a minute on first load…"` (cold-start awareness pattern from Phase 6 D-19)
  - shadcn Skeleton chips below the status copy (8 placeholder chips)
  - On success: replace skeletons with the actual diff chips

### G. Langfuse observability (PROF-06 fifth criterion)

- **D-32: Single Langfuse trace per upload, multi-span structure.** Trace name = `resume_upload`. Spans (Langfuse `@observe()` decorators or manual `langfuse.start_span()`):
  1. `text_extract` — metadata: `file_type`, `file_size_bytes`, `page_count` (PDF only), `char_count`, `latency_ms`
  2. `llm_extract` — metadata: `model`, `prompt_version`, `input_tokens`, `output_tokens`, `cost_usd`, `latency_ms`; Langfuse auto-captures this if Instructor's OpenAI client is wrapped with `langfuse.openai`
  3. `diff_compute` — metadata: `added_count`, `removed_count`, `unchanged_count`, `latency_ms`
  4. `profile_save` — fires ONLY if `PATCH /profile` is called with the matching `extraction_id` within 10 minutes of the upload (trace correlation via `extraction_id` field on both endpoints — D-13, D-21). Metadata: `written_skill_count`, `latency_ms`.

  Trace correlation: `POST /profile/upload` calls `langfuse.trace(name="resume_upload", id=str(extraction_id), user_id=str(user_id))`; `PATCH /profile` calls `langfuse.trace(id=str(extraction_id))` to attach to the same trace if the ID was provided. Counterfactual: separate traces — rejected, REQ-PROF-06 success-criterion 5 is literal: "single trace per upload spanning… PATCH".

- **D-33: PII policy — raw resume text NEVER traced.** Langfuse traces hold only metadata + extracted structured data. The `text_extract` span's `output` field is `{"char_count": N, "page_count": M}`, NOT the text itself. The `llm_extract` span's input field omits the resume body (or replaces it with `"[REDACTED — char_count=N]"`); the output is the structured `ResumeExtraction` JSON, which contains skill names and salary (these ARE traced because they're the operational signal — and skill names alone don't constitute PII). Future shareable trace dashboards are safe. Counterfactual: trace everything for full audit — rejected, resumes contain name + contact + employer history.

- **D-34: Langfuse `userId` = `str(seeded_user_id)`.** Future multi-user makes this automatic. v1: same value every upload. Trace `tags` = `["resume", "phase-7"]` for dashboard filtering. Trace `release` = current git short SHA (matches existing observability convention if set; otherwise omit).

### H. Error taxonomy

- **D-35: Errors return `HTTPException(status, detail={reason, message})` with a structured detail dict.** Reasons:
  | Status | Reason | When |
  |--------|--------|------|
  | 413 | `file_too_large` | Content-Length > 2_000_000 (D-07 pre-body check) |
  | 415 | `unsupported_file_type` | Extension or Content-Type fails whitelist (D-08) |
  | 422 | `pdf_encrypted` | `pypdf.errors.PdfReadError` raised |
  | 422 | `text_extraction_failed` | Parsed text < 100 non-whitespace chars (D-10) |
  | 422 | `extraction_failed` | Instructor exhausted 3 retries (D-16) |
  | 422 | `empty_skills` | Extraction returned 0 skills (likely off-topic file — résumé of a non-engineer, college essay, etc.) |
  | 503 | `llm_unavailable` | OpenAI API outage (specific `openai.APIError` subclasses) |

  Frontend `useResumeUpload` maps `reason` → user-facing copy via a small object literal:
  ```typescript
  const COPY: Record<string, {title: string; body: string}> = {
    file_too_large: { title: "File too large", body: "Resume must be ≤2 MB." },
    unsupported_file_type: { title: "Unsupported format", body: "Upload a PDF or DOCX." },
    pdf_encrypted: { title: "Encrypted PDF", body: "Remove the password and try again." },
    text_extraction_failed: { title: "Couldn't read the file", body: "The file appears to be a scanned image. v1 doesn't support OCR." },
    extraction_failed: { title: "Extraction failed", body: "The agent couldn't parse the resume. Try again or simplify the formatting." },
    empty_skills: { title: "No skills found", body: "The extracted skill list is empty. Is this a resume?" },
    llm_unavailable: { title: "Service unavailable", body: "The LLM is down. Please try again in a few minutes." },
  }
  ```
  Backend logs structured: `log.warning("resume_upload_failed", reason=..., file_type=..., file_size_bytes=...)`.

- **D-36: Test coverage = happy path + 5 error paths in `tests/test_profile.py` (NEW).** Test cases:
  1. Happy path: valid PDF → 200 + diff present
  2. Happy path: valid DOCX → 200 + diff present
  3. 413: oversized file via `Content-Length` header
  4. 415: `.txt` file rejected
  5. 422: encrypted PDF (use fixture `tests/fixtures/encrypted-sample.pdf`)
  6. 422: empty-text PDF (image-only fixture)
  7. 422: Instructor raises after 3 retries (mock `extract_resume` to raise `ValidationError`)

  AUTH-06 single-user guard inherited automatically via `Depends(get_current_user_id)`. Resume extraction tests live in `tests/test_resume_extractor.py` covering prompt rejection rules + structured output validity (mirror of `tests/test_extraction.py`).
