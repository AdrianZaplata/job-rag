> Visual and interaction contract for the Profile & Resume Upload surface (current-profile read → resume upload → extraction-progress → reviewable skill diff with status pills → tick/edit/save). Generated by gsd-ui-researcher; revised against CONTEXT.md.
>
> **Inheritance principle.** Phase 7 inherits the full Phase 4 design system verbatim — spacing scale, typography ramp, color tokens, motion rules, accessibility floor, copywriting tone, registry safety. It also inherits the per-widget loading/empty/error layered-state pattern Phase 5 established (`<Alert variant="destructive">` for surface errors, `<EmptyState>` for no-data). >
> If any value in this document conflicts with `04-UI-SPEC.md`, **Phase 4 wins** — this is a corollary contract, not a redefinition.
>
> **Revision note (rev 1).** A prior auto-mode draft ran without CONTEXT.md visibility and invented several patterns (ConfirmDestructiveDialog, three-bucket SkillDiffPanel layout, ProfileSaveBar as a separate component, CurrentProfileCard + Delete-resume affordance, asymmetric per-bucket default selection semantics, net-new `checkbox` + `label` installs). **Phase 7 spacing exceptions** (2 documented exceptions; not part of the prescriptive token set):

- **Chip row vertical padding:** `py-1.5` (6px). Tight rows are required to hit the density target — see §14 below (target: review ~50-80 chips in a viewport without excessive scrolling). Standard `py-2` (8px) at 14px line-height yields ~30px rows; `py-1.5` yields ~27px. Used ONLY for chip rows; not propagated as a new token. WCAG 2.5.5 touch-target floor is satisfied because the row's interactive controls (checkbox + edit pencil) each carry their own `h-9 w-9` ghost-button hit-area which OVERLAPS the row but is not constrained by `py-1.5`.
- **Selected-file pill horizontal padding:** `px-3` (12px). One-off pill chrome; standard `px-2` (8px) reads cramped around the FileText icon + filename + size + button cluster. Used ONLY for the selected-file pill in `<ResumeUploader>`; not propagated.

**Drop-zone padding scale** `p-8 md:p-12` (32/48px) is NOT an exception — both ARE Phase 4 §2 tokens; the responsive bump is per-surface only.

All other values not in the Phase 4 §2 table or above (e.g., `p-3`, `p-5`, `gap-3`, `gap-10`) are **disallowed** in profile-feature files without an explicit override note in the PR.

### Phase 7 additions to Phase 4 §4 "Accent reserved-for list" (extends, does not replace)

Phase 4 §4 reserved the accent (`--primary`) for four categories:
1. Primary CTA buttons (one per surface)
2. Active nav-tab indicator
3. Focus rings (`--ring`)
4. Theme-toggle hover dot (sub-pixel)

**Phase 7 adds two entries under category 1** (which still permits one Primary CTA visible at any given moment — the two entries are mutually-exclusive across surface states):

1. **Upload button** in `<ResumeUploader>` (label "Upload resume"). Visible ONLY when a file is selected but not yet uploaded. Goes away once extraction begins. NEVER co-renders with #2.
2. **Save button** in `<ReviewPanel>` CardFooter. Visible ONLY when `state === 'reviewing'` (i.e., after a successful upload response). NEVER co-renders with #1 because the diff panel only appears AFTER successful extraction (which dismisses the `<ResumeUploader>`).

At no point does the surface render two accented buttons simultaneously.

### Destructive token usage

Phase 7 uses `--destructive` ONLY for inline error Alerts (upload-failure / extraction-failure / save-failure). NO destructive button is rendered anywhere in Phase 7 — there is no "Delete resume" surface (CONTEXT has no DELETE endpoint; the discard action is `variant="outline"` per D-27, and clicking Discard simply resets the in-memory draft + flips state back to `idle` without backend mutation).

This is a deliberate departure from the prior auto-mode draft: that draft invented a "Delete resume" + "first destructive confirmation dialog in the app" pattern that CONTEXT does NOT endorse. CONTEXT D-Discretion shows Phase 7's only destructive-adjacent affordances are (a) "Remove" on the selected-file pill (just clears `state.file`; no backend) and (b) "Discard" on the review panel (just flips state back to `idle`; no backend). Neither requires a confirmation dialog.

### Color-blind safety

Phase 7 NEVER uses color alone to convey meaning:
- **Status pills:** distinguished by literal text label (`+ ADDED` / `− REMOVED` / `= UNCHANGED`) — the prefix glyph (`+` / `−` / `=`) reinforces the semantic for monochrome rendering / screen readers.
- **Selected vs unselected skills:** distinguished by checkbox `data-state` (or native `checked` attribute) — NOT by row color.
- **Edited skill names:** distinguished by `(edited)` text tag next to the name — NOT by row color.
- **Errors:** the `<Alert>` carries both red color AND an `AlertCircle` icon AND descriptive copy.

**Client-side validation (D-30 pre-check; backend D-07 + D-08 are authoritative):**

```ts
function validateFile(file: File): UploadError | null {
  const MAX_BYTES = 2_000_000  // matches backend D-07 MAX_RESUME_SIZE_BYTES = 2_000_000
  const ALLOWED_EXTS = ['.pdf', '.docx']
  const ALLOWED_TYPES = [
    'application/pdf',
    'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
  ]

  if (file.size > MAX_BYTES) {
    return { reason: 'file_too_large', context: { size: formatBytes(file.size) } }
  }
  const extOk = ALLOWED_EXTS.some(ext => file.name.toLowerCase().endsWith(ext))
  const typeOk = ALLOWED_TYPES.includes(file.type)
  if (!extOk && !typeOk) {
    return { reason: 'unsupported_file_type' }
  }
  return null
}
```

Saves a round-trip vs the backend; backend remains authoritative (D-07 pre-body check + D-08 extension + Content-Type intersection).

### 6f. useResumeUpload (hook)

**Location:** `frontend/src/components/profile/useResumeUpload.ts`

**Purpose:** Single hook that owns the entire upload + review lifecycle per CONTEXT D-29. Wraps a TanStack `useMutation` for the upload AND for the save; owns the state machine (`idle` ↔ `reviewing`); manages the cold-start status-copy timers per D-31.

**Public API (locked shape; executor implements):**

```ts
type ResumeUploadHook = {
  // Phase machine
  phase: 'idle' | 'reviewing'

  // Idle-state surface (consumed by <ResumeUploader>)
  uploadState: 'idle' | 'file-picked' | 'uploading' | 'extracting' | 'error'
  file: File | null
  error: UploadError | null
  statusCopy: string  // D-31 derived from elapsed-time timer
  selectFile: (file: File) => void
  clearFile: () => void
  upload: () => void

  // Reviewing-state surface (consumed by <ReviewPanel>)
  diff: SkillDiffItem[] | null
  extractionSummary: { n_extracted: number; n_added: number } | null
  extractionId: string | null  // UUID for Langfuse correlation per D-21
  isSaving: boolean
  saveError: { reason: string; message?: string } | null
  discardReview: () => void
  saveReview: (finalSkills: UserSkill[]) => void
}
```

**Internal state machine:**

```
idle ────selectFile────► idle (file set)
idle ────upload────────► uploading ──ack──► extracting ──response──► reviewing (diff loaded)
                                          ──error──────► idle (error set)
                            ──error──────► idle (error set)

reviewing ──discardReview─────► idle (diff cleared; error cleared)
reviewing ──saveReview────────► reviewing (isSaving=true) ──success──► idle (toast; cache invalidated)
                                                          ──error────► reviewing (saveError set; toast)
```

**Cold-start timer per D-31 (mirrors Phase 6 §12 `useChatStream` pattern):**

1. On `upload()` call: clear any existing timers; set `statusCopy = "Reading your resume…"`; arm `setTimeout(2000)` to flip `statusCopy = "Asking the agent to extract skills…"`; arm `setTimeout(10000)` to flip `statusCopy = "Still working — extraction can take a minute on first load…"`.
2. On extraction response (success OR error): clear both timers.
3. On `<Profile>` unmount: cleanup runs `abortController.abort()` + clears timers.

**Mutation patterns:**

- Upload mutation: `mutationFn: (file: File) => uploadResume(file, signal)`. On success: store `data` (the `ResumeUploadResponse`), flip `phase = 'reviewing'`. On error: parse the response body's `{reason, message}` (D-35) into `error` state; stay in `idle`.
- Save mutation: `mutationFn: (payload: UserProfileUpdate) => saveProfile(payload, signal)`. On success: `queryClient.invalidateQueries({queryKey: ['profile']})` + `queryClient.invalidateQueries({queryKey: ['dashboard']})` (per CONTEXT "Claude's Discretion" verbatim); fire `toast.success("Profile saved")`; flip `phase = 'idle'`. On error: fire `toast.error("Save failed: {message}")`; set `saveError`; stay in `reviewing`.

**`extractionId` correlation per D-32:**

The upload response's `extraction_id` (server-generated UUID, D-13) is captured into the hook's state. On save, it's included in the PATCH payload's `extraction_id` field (D-21) so the backend can attach the `profile_save` Langfuse span to the same trace. If the user discards without saving, no PATCH fires; the upload's Langfuse trace lives without a `profile_save` span. CONTEXT D-32 documents this is a within-10-minute correlation window.

## 11. Default Selection Semantics (LOCKED per D-25)

The diff panel's per-chip default checkbox state is a load-bearing UX decision. CONTEXT D-25 LOCKS it:

| `source` | Default `checked` | Semantic |
|----------|-------------------|----------|
| `added` | `true` (✓) | "Trust the extraction by default; user un-ticks noise." Minimizes the action gradient for the happy-path user who uploaded an accurate resume. |
| `removed` | `false` (☐) | "Do NOT keep removed skills unless the user explicitly opts in." The chip's checkbox semantic in the `removed` bucket is "rescue this skill from removal" — leaving unchecked drops it; checking keeps it. |
| `unchanged` | `true` (✓) | "Existing skills stay by default." The user reviews them only if they want to audit; otherwise unchanged skills are silently kept. |

This is D-25 verbatim. The default state is: `Save (3 new · 0 keep removed · 47 unchanged)` — the user can save with one click and the result is "replace current profile with the extracted set, accepting all additions, accepting all removals, keeping all unchanged".

**Save payload contract:**

When the user clicks Save, the PATCH `/profile` body's `skills` field is the union of:
- All CHECKED `added` items (using `editedName` if non-null per D-26, else `name`).
- All CHECKED `unchanged` items (using `name` — no edit per D-26).
- All CHECKED `removed` items (rescued — kept; using `name` — no edit per D-26).

UN-CHECKED `removed` items are absent from the payload → backend drops them. UN-CHECKED `added` items are absent → backend never gets them. The PATCH body is the AUTHORITATIVE "what the user wants the profile to be after save" — full replace per backend D-21.

**Empty save guard:**

If the resulting skill union is empty (user unchecked everything across all three statuses), the Save button is disabled with the tooltip `No skills selected. Saving would clear your profile.` The user cannot accidentally PATCH an empty skills list. To intentionally clear all skills, the user must un-check every chip — the empty save is reached deliberately, not slipped into.

## 14. Density & Reading Order

Phase 7 density target: with a typical resume diff (~5 added, ~2 removed, ~40 unchanged ≈ 47 chips), the user should be able to:

- See the CardHeader summary (`Found 47 skills. 5 new vs your current profile.`) without scrolling.
- See the Save button live-count summary in the sticky footer at all times.
- Scroll through the chip list inside the `max-h-[60vh] overflow-y-auto` CardContent.

| Element | Vertical cost |
|---------|---------------|

| Page h1 + padding | ~40px |

1. Top: AppShell top-nav.
2. Below: page h1.
3. ReviewPanel CardHeader (extraction summary — context).

1. AppShell top-nav.

3. ProfileView (read what you currently have).
