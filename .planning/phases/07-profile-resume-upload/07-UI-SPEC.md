# Phase 7 — UI Design Contract

> Visual and interaction contract for the Profile & Resume Upload surface (resume upload → extraction-progress → reviewable skill diff → tick/edit/save). Generated by gsd-ui-researcher in --auto mode.
>
> **Inheritance principle.** Phase 7 inherits the full Phase 4 design system verbatim — spacing scale, typography ramp, color tokens, motion rules, accessibility floor, copywriting tone, registry safety. It also inherits the per-widget loading/empty/error layered-state pattern Phase 5 established (`<Alert variant="destructive">` for surface errors, `<EmptyState>` for no-data). This document only declares what is NET-NEW: the `/profile` route content, the `components/profile/` feature folder, the upload affordance, the extraction-progress UX, the diff panel + skill chip / row interactions, the save flow, the file-too-large / wrong-type / extraction-failure / save-failure error states, and the destructive "delete uploaded resume" confirmation pattern (FIRST destructive confirmation surface in the app).
>
> If any value in this document conflicts with `04-UI-SPEC.md`, **Phase 4 wins** — this is a corollary contract, not a redefinition.
>
> Phase 7 is the FIRST write-back surface in the app. Prior phases (Dashboard, Chat) are read-only. The diff-review + chip-editing + save pattern Phase 7 lands here is the canonical pattern future write-back surfaces will compose.

**Phase 7 spacing exceptions** (3 documented exceptions; not part of the prescriptive token set):

- **Save bar vertical padding:** `py-3` (12px). Inherits the documented exception from `06-UI-SPEC.md` §3 (composer container). Same justification: smallest comfortable vertical breathing room around an `h-9` button on a sticky-bottom action bar. NOT propagated as a new token.
- **Skill row vertical padding:** `py-1.5` (6px). Tight rows are required to hit the density target — see §15 below (target: 8-10 skill rows visible per bucket in a typical viewport). Standard `py-2` (8px) at 14px line-height yields ~30px rows; `py-1.5` yields ~27px. Used ONLY for skill rows; not propagated as a new token. WCAG 2.5.5 touch-target floor is satisfied because the row's interactive controls (checkbox + edit + remove menu) each carry their own `h-9 w-9` ghost-button hit-area which OVERLAPS the row but is not constrained by `py-1.5`.
- **Drop-zone padding scale:** `p-8 md:p-12` (32/48px). Both ARE Phase 4 §2 tokens — listed here only because the responsive bump is per-surface. Drop-zone needs generous interior to feel "drop here", but on mobile (< 768px) `p-12` would push the affordance below the fold.

All other values not in the Phase 4 §2 table or above (e.g., `p-3`, `p-5`, `gap-3`, `gap-10`) are **disallowed** in profile-feature files without an explicit override note in the PR.

### Phase 7 additions to Phase 4 §4 "Accent reserved-for list" (extends, does not replace)

Phase 4 §4 reserved the accent (`--primary`) for four categories:
1. Primary CTA buttons (one per surface)
2. Active nav-tab indicator
3. Focus rings (`--ring`)
4. Theme-toggle hover dot (sub-pixel)

**Phase 7 adds two entries under category 1** (which still permits one Primary CTA visible at any given moment — the two entries are mutually-exclusive across surface states):

1. **Upload button** in `ResumeUploadCard` (label "Upload resume"). Visible ONLY when a file is selected but not yet uploaded. Goes away once extraction begins. NEVER co-renders with #2.
2. **Save button** in `ProfileSaveBar`. Visible ONLY when the diff panel is open AND the draft is dirty (at least one tick or edit relative to extraction output). NEVER co-renders with #1 because the diff panel only appears AFTER successful extraction (which dismisses the upload affordance).

At no point does the surface render two accented buttons simultaneously.

### Phase 7 additions to Phase 4 §4 "Destructive reserved-for list" (extends, does not replace)

Phase 4 §4 reserved `--destructive` for sign-out menu item + (pre-allocated) ErrorBoundary "Reload page" (never rendered). Phase 6 added Stop button + inline error Alerts. Phase 7 adds:

1. **"Delete resume" button** in `CurrentProfileCard` — destructive variant, renders ONLY when a resume has been uploaded previously and persisted. Triggers `ConfirmDestructiveDialog` before performing the delete (see §6g).
2. **"Discard changes" button** in `ProfileSaveBar` — `variant="ghost"` styling (NOT destructive fill — the action is reversible until save; ghost is sufficient). Triggers `ConfirmDestructiveDialog` ONLY when the draft has actual edits (otherwise it's a no-op silent revert). Note: button styling is ghost but the confirmation flow uses the destructive-confirmation pattern because it loses unsaved work.
3. **Confirm-action button INSIDE the `ConfirmDestructiveDialog`** for "Delete resume" — `variant="destructive"` fill (red). The "Discard changes" confirmation uses `variant="default"` because the underlying action is reversible (just clears the in-memory draft).
4. **Inline error Alerts** for upload-failure / extraction-failure / save-failure (`variant="destructive"`).

No other element in Phase 7 uses `--destructive`. Specifically: the "removed" bucket items do NOT render in destructive color — they render with `line-through` on `--muted-foreground` (information, not action). Removing a skill is a tick-action on the user's part, not an irreversible system delete.

### Color-blind safety

Phase 7 NEVER uses color alone to convey meaning:
- **Diff buckets:** distinguished by section heading (`Added` / `Removed` / `Unchanged`) + bucket icon (`Plus` / `Minus` / `Equal` from lucide) + position. Badge color is uniform `--secondary` across all three buckets.
- **Selected vs unselected skills:** distinguished by `<Checkbox>` data-state (checked/unchecked) — NOT by row color.
- **Edited skill names:** distinguished by `Pencil` icon next to the name (and a `text-xs text-muted-foreground` "(edited)" suffix) — NOT by row color.
- **Errors:** the `<Alert>` carries both red color AND an `AlertCircle` icon AND descriptive copy.

**Client-side validation (`validateFile`):**

```ts
function validateFile(file: File): UploadError | null {
  const MAX_BYTES = 2 * 1024 * 1024  // 2 MB
  const ALLOWED_TYPES = [
    'application/pdf',
    'application/vnd.openxmlformats-officedocument.wordprocessingml.document',  // .docx
  ]
  const ALLOWED_EXTS = ['.pdf', '.docx']

  if (file.size > MAX_BYTES) {
    return {
      kind: 'too-large',
      message: `Resume must be 2 MB or less. Yours is ${formatBytes(file.size)}.`,
    }
  }
  const extOk = ALLOWED_EXTS.some(ext => file.name.toLowerCase().endsWith(ext))
  const typeOk = ALLOWED_TYPES.includes(file.type)
  if (!extOk && !typeOk) {
    return {
      kind: 'wrong-type',
      message: `Resume must be a PDF or DOCX file. Yours appears to be ${file.type || 'an unknown type'}.`,
    }
  }
  return null
}
```

Client-side validation is a UX layer; backend MUST also enforce `413 Payload Too Large` and content-type check (PROF-02 explicit). Client validation prevents the round-trip but is not authoritative.

  DialogDescription,

  onOpenChange,
  title,
  description,
  confirmLabel,
  cancelLabel = 'Cancel',
  variant = 'destructive',
  isPending = false,
  onConfirm,
}: ConfirmDestructiveDialogProps) {
  return (
    <Dialog open={open} onOpenChange={(next) => !isPending && onOpenChange(next)}>
      <DialogContent className="max-w-md">
        <DialogHeader>
          <DialogTitle>{title}</DialogTitle>
          <DialogDescription>{description}</DialogDescription>
        </DialogHeader>
        <DialogFooter>

            disabled={isPending}
          >

          </Button>

            variant={variant}
            onClick={onConfirm}
            disabled={isPending}
            autoFocus={false}  /* DO NOT autofocus the destructive button — see below */

**Dialog contract:**

| Property | Value |
|----------|-------|
| Container width | `max-w-md` (28rem / 448px) — narrower than info dialogs (`max-w-2xl`) to feel weightier and force user attention |
| Title typography | shadcn DialogTitle default — `text-lg font-semibold` (18px) |
| Description typography | shadcn DialogDescription default — `text-sm text-muted-foreground` (14px muted) |
| Cancel button | Footer-LEFT, `variant="ghost"`, default label `Cancel` |
| Confirm button | Footer-RIGHT, `variant="destructive"` (or `default` for reversible variants), explicit verb+noun label (e.g. `Delete resume`, `Discard changes`) |
| Initial focus | Cancel button (NOT Confirm) — see "Autofocus rationale" below |
| Keyboard close | `Escape` closes (Radix Dialog default) → fires `onOpenChange(false)` |
| Overlay click | Closes (Radix Dialog default) — only if `!isPending` (locked during async work) |
| Focus trap | Yes (Radix Dialog default) |
| Backdrop dismiss while pending | Disabled — `onOpenChange={(next) => !isPending && onOpenChange(next)}` prevents the user from clicking-out mid-fetch |

**Autofocus rationale (LOCKED):**

The Cancel button receives initial focus (via Radix Dialog default behavior — first focusable element is `<Cancel>`). The destructive Confirm button does NOT autofocus. Rationale:

1. **Slip-resistance.** Spacebar / Enter immediately after the Dialog opens fires the focused button. Autofocusing Confirm would mean a careless Enter from the keyboard-driven user destroys the resume. Autofocusing Cancel means the same keystroke bails out — safe-by-default.
2. **WCAG 2.4.7 Focus Visible** is satisfied because the focus ring renders on Cancel.
3. **Linear / GitHub / Apple precedent** — destructive dialogs across these tools also focus the safe button.

This is a Phase 7 LOCKED contract — future destructive surfaces (and the Phase 8 "delete all my data" if it ever ships) MUST follow it.

3. The `coldStart` flag is passed to `<ExtractionProgress>` which swaps the step 2 label.

## 11. Default Selection Semantics (LOCKED)

The diff panel's per-bucket default checkbox state is a load-bearing UX decision. Phase 7 locks it here:

| Bucket | Default checkbox state | Semantic |
|--------|----------------------|----------|
| `added` | ALL CHECKED | "Trust the extraction by default; user un-ticks noise." This minimizes the action gradient for the happy-path user who uploaded an accurate resume. |
| `removed` | ALL UN-CHECKED | "Do NOT drop existing skills unless the user explicitly opts in." This is the safety-by-default choice — accidentally clicking Save with the removed bucket unreviewed would silently delete skills the user has previously confirmed. Forcing an explicit tick to rescue (the inverse of "drop") would be conceptually noisier than "un-checked means safe, checked means I want this one back". |
| `unchanged` | ALL CHECKED (and bucket collapsed) | "Existing skills stay by default." The user sees them only if they want to audit; otherwise the unchanged bucket is silently confirmed. |

**Save payload contract:**

When the user clicks Save, the PATCH body contains the union of:
- All CHECKED `added` skills (using their `editedName` if edited, else original `name`).
- All CHECKED `unchanged` skills (using `editedName` or `name`).
- All CHECKED `removed` skills (rescued — kept; using `editedName` or `name`).

UN-CHECKED `removed` skills are absent from the payload → backend drops them. UN-CHECKED `added` skills are absent → backend never gets them. The PATCH body is the authoritative "what the user wants the profile to be after save" — a full replacement of the `skills` list, not a delta.

**Validation:**

If the union is empty (`selectedCount === 0`), Save is disabled (status text shows `No skills selected`). The user cannot accidentally PATCH an empty skills list. To intentionally clear all skills, the user must un-check every row across all three buckets — the empty save-payload is reached deliberately, not slipped into.

**Why explicit per-bucket asymmetry (not "all-checked across the board"):**

The two destructive directions (dropping a skill vs adding a skill) have different reversibility profiles:

- Adding a noisy skill: user notices in the dashboard CV-vs-market view, re-uploads, un-ticks. Recovery = "do the next upload differently."
- Dropping a skill the user actually has: user notices the skill is missing, re-uploads, but the resume may not even contain that skill if it was a manual-only entry. Recovery = "manually re-add" (NOT a Phase 7 feature). Asymmetric pain.

So the asymmetric default surfaces this: added defaults to confirmed (low-stakes); removed defaults to safe (high-stakes opt-in to actually drop).

## 14. Density & Reading Order

- See the bucket header summary (`+15 added · −3 removed · 40 unchanged`) without scrolling.
- See ALL `added` and `removed` rows without scrolling (these are the action items).
- Have the `unchanged` bucket collapsed by default (audit-on-demand).

| 6 | Inline-edit affordance: pencil icon vs double-click vs always-visible | **Pencil icon button (visible on row hover and always on focus)** | Pencil is the universal "edit" affordance; visible on hover keeps the row tidy when idle; visible on focus ensures keyboard-only users can reach it. Double-click would be a hidden gesture (poor a11y); always-visible would clutter the row. |
| 7 | Edit commit on Enter vs Tab vs explicit Save button | **Enter commits, Escape cancels, Save/Cancel buttons also present** | Multiple paths to the same action (keyboard + mouse), all redundant by design. Tab moving forward would risk losing edits when navigating to the next row; Tab keeps default focus-move behavior but does NOT commit (avoid surprises). |
| 8 | "Remove from list" placement: button in row vs overflow menu vs swipe | **Overflow menu (3-dot `MoreHorizontal` trigger → DropdownMenu)** | Per-row remove is a secondary destructive-ish action. A visible Remove button would compete with the Pencil and create row clutter; overflow menu hides it behind one tap/click while keeping it discoverable. Matches GitHub PR comment pattern. |
| 9 | Save button position: top-right of diff card vs sticky-bottom bar | **Sticky-bottom bar** | Diff is potentially long (108+ skills); top-right Save would scroll off. Sticky-bottom matches Phase 6 ChatComposer pattern; establishes Phase 4+ convention for primary-action-at-viewport-bottom on work-style routes. |
| 10 | Discard button styling: destructive variant vs ghost | **Ghost** | The action triggers a confirmation dialog (which can use a destructive variant if needed); inline destructive button styling for a reversible-with-confirmation action would over-signal. Ghost + confirm-dialog is the canonical "second-class destructive" pattern. |
| 11 | Cold-start UX in extraction phase | **Reuse Phase 6 §12 pattern verbatim** (10s threshold, "Warming up the agent — this can take ~4 minutes after idle") | Same backend (ACA scale-to-zero). Same UX problem. Same solution. Adapted to step-label: `Extracting skills — this can take ~4 minutes after idle…`. |
| 12 | Navigation-away guard | **`useBlocker` from React Router 7 + ConfirmDestructiveDialog** | The browser's native `beforeUnload` only fires on hard-close/refresh; React Router's `useBlocker` catches in-app navigation. We need both: `useBlocker` for nav-tab clicks + native `beforeunload` event handler for browser close (set + remove via useEffect). Both reuse the same ConfirmDestructiveDialog component. |
| 13 | File MIME type validation: client-side strict vs lenient | **Strict on extension + lenient on MIME type** | Browsers sometimes report empty / mistyped MIME (`""` for DOCX on Safari). Extension is the more reliable signal; MIME is a backup check. If EITHER passes → accept. Backend validates authoritatively. |
| 14 | Upload progress bar (`<Progress>` shadcn primitive) | **NO in v1** — single-step Skeleton pulse in ExtractionProgress | The upload phase itself is sub-second on modern connections for a 2 MB file; the bulk of perceived latency is the LLM extraction round-trip, which is non-streamable (Instructor returns the full result; no partial progress). A fake progress bar would lie. Skeleton pulse honestly says "working". Re-evaluate in Phase 8 if streaming extraction lands. |
| 15 | "Edited" tag visibility | **Always visible next to skill name when `editedName !== null`** | Communicates the audit signal without requiring a hover; matches Linear's "edited" tag on comments. Format: `<span className="text-xs text-muted-foreground ml-2">(edited)</span>`. |
| 16 | Empty-bucket display: hide bucket entirely vs render with "No skills in this bucket." | **Render with note** | Hiding an entire bucket would make the user wonder if it's broken; rendering with a zero count visible (in the badge cluster) + empty body with a soft note is the honest signal. |
| 17 | Diff bucket count badge color | **Uniform `--secondary`** (NOT colored per bucket) | Color-blind safety; the bucket label is the semantic. Coloring added=green and removed=red would conflate "added is good" / "removed is bad" — but a removed skill might be GOOD (the user no longer wants jQuery on their profile). Neutral coloring lets the user make the judgment. |
| 18 | Per-skill confidence/category display | **NOT in v1** | The Instructor extraction may return `category` and confidence scores; v1 ignores them in the UI. Future: tooltip or sub-line. Keeps row density honest. |
| 19 | Bulk-edit affordance (e.g. "edit all `aws*` to `Amazon Web Services`") | **NOT in v1** | Alias resolution is a v2 platform feature (REQUIREMENTS PLAT2-04). Phase 7 ships per-row edit only. |

## 19. Things This Phase Does Not Decide

Out-of-scope for Phase 7 UI-SPEC:

### Profile-page features deferred

- Editing target roles, preferred locations, or min-salary fields in v1 (skills-only edit per PROF-05/PROF-06 — the other fields stay seeded). Phase 8 polish window may add.
- Skill category override (user picks `hard`/`soft`/`domain` per skill).
- Skill ordering / priority weighting.
- Multiple resume versions (only one "current" resume in v1; uploading replaces).
- Resume preview / inline PDF render — out of scope; the diff IS the actionable artifact.
- Export profile as JSON / CV markdown — deferred.

### Surface affordances deferred

- Drag-and-drop reorder of skills within a bucket.
- Bulk inline-edit (e.g., select 5 rows, rename pattern).
- Search/filter within the skill list (Cmd+F is the v1 path).
- Per-bucket "X / Y selected" sub-count in the bucket header (status text in ProfileSaveBar shows the total; per-bucket is a sub-feature).

### Error-recovery deferred

- Auto-save draft to localStorage so refresh doesn't lose review state (v1 matches CHAT-06 single-turn pattern — refresh loses).
- "Restore my last saved profile" undo button after a save (the user can re-upload to redo).

### Other surfaces

- Chat (Phase 6).
- Dashboard (Phase 5).
- Cold-start "Always-warm ACA" mitigation — Phase 8 portfolio polish (€8/mo to eliminate).

### Performance / infra

- The diff panel's row rendering is plain map; if the unchanged bucket explodes to 500+ skills in some future user's profile, virtualization would help. v1 corpus's profile rarely exceeds 100 skills; not virtualizing.
