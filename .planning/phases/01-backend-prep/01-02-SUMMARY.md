## Decisions Made

- **Three-migration split (0001 baseline pre-Phase-1 schema, 0002 users + user_profile + seed, 0003 career_id) instead of one all-up baseline.** Required because the dev DB has only the pre-Phase-1 3-table schema; stamping it at "head" of an all-up baseline would skip the users/user_profile/career_id creation and contradict must-have truth #4 (career_id backfilled on all pre-existing rows). Splitting lets dev stamp at 0001 (matches its actual current state) and run 0002+0003 cleanly.
- **server_default= on user_profile JSON columns instead of Python-side default=.** Plan PATTERNS.md §File 6 specified Python-side `default="[]"` but the migration spec used `server_default="[]"` — a real conflict. `alembic check` reports drift on every column unless they match. server_default is the more useful path (DDL-side defaults apply to direct INSERTs without ORM, future-proof for admin/external tooling).
- **Direct INSERT INTO alembic_version for dev DB stamp instead of `alembic stamp 0001`.** Adrian's dev DB password contains `&%!$` which require URL-encoding to `%26%21%24` etc. Alembic's `set_main_option` invokes ConfigParser interpolation which interprets `%` as the start of a substitution and chokes on the encoded password. Direct SQL bypasses the parser. Future migrations on dev (after stamping) use `DATABASE_URL='postgresql://postgres:%%26%%21%%24...' alembic upgrade head` with `%%` to escape interpolation.
- **socat sidecar (alpine/socat:latest) for host→dev-DB access.** docker-compose.yml only `expose:`'s the dev DB to the docker network, no host port mapping. Rather than modify compose (which is committed config), spin up a one-shot socat container on `--network job-rag_default` mapping host `5436` → `job-rag-db-1:5432`. Stop the container after stamp+upgrade. Ephemeral; no compose changes.
