from job_rag.db import models as _models  # noqa: F401 side-effect import

config = context.config
if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata


def run_migrations_online() -> None:
    config.set_main_option("sqlalchemy.url", os.environ["DATABASE_URL"])
    connectable = engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    with connectable.connect() as connection:
        connection.dialect.ischema_names["vector"] = pgvector.sqlalchemy.Vector
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
            compare_type=True,
            compare_server_default=True,
        )
        with context.begin_transaction():
            context.run_migrations()

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Alembic env.py → DB | Migration scripts run with app user creds from DATABASE_URL; boundary is the DB's CREATE privilege for CREATE EXTENSION |
| SEEDED_USER_ID config → migration seed | Single source of truth (settings.seeded_user_id) crosses from Python config into the DDL; any drift is an attack surface |
| Alembic env.py file → CI job | If env.py is modified to log DATABASE_URL, secrets leak in CI logs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | alembic/env.py + 0001_baseline.py `CREATE EXTENSION vector` | mitigate | Local dev Postgres runs as superuser (docker-compose convention) — CREATE EXTENSION succeeds. Production (Phase 3 Azure): the extension is pre-created by the DBA (or via `azure.extensions`), and the app user does NOT have CREATE EXTENSION privilege — `IF NOT EXISTS` makes the migration a no-op. Phase 3 planner carries the responsibility of pre-enabling the extension. Per Pitfall 9. |
| T-02-02 | Tampering | alembic/versions/0002_add_user_profile.py (SEEDED_USER_ID INSERT) | mitigate | Seed INSERT uses `ON CONFLICT (id) DO NOTHING` — re-running cannot overwrite existing row. SEEDED_USER_ID is imported from `settings.seeded_user_id` — single source of truth; any drift between config and migration fails a grep guard in Plan 01's test_alembic.py. |
| T-02-03 | Tampering | alembic/versions/0002_add_user_profile.py user_profile.user_id column | mitigate | Column has NO `server_default` and NO `default` — enforced by Plan 01's grep-guard test (`tests/test_alembic.py::test_no_default_uuid_on_user_id_columns`). Pitfall 18 (silent multi-user collision) eliminated. |
| T-02-04 | Information Disclosure | alembic/env.py DATABASE_URL | mitigate | `config.set_main_option("sqlalchemy.url", os.environ["DATABASE_URL"])` reads from env; `alembic.ini` leaves `sqlalchemy.url =` blank so the connection string is not committed. Alembic logger level WARN by default — URL not logged. |
| T-02-05 | Denial of Service | alembic/env.py NullPool | accept | NullPool opens transient connections; if CLI + API + alembic concurrent, B1ms cap hit (Pitfall I). Phase 3 runbook enforces ordering; local dev single-developer, irrelevant. |
