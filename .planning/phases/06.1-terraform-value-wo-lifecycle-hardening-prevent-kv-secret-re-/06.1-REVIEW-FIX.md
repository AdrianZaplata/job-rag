## Fixed Issues

### CR-01: CI pre-apply guard fails OPEN on every `az containerapp show` error class except first-apply

**Files modified:** `.github/workflows/deploy-infra.yml`
**Commit:** `f48c45a` (combined with WR-04 — same file, same shell block)
**Applied fix:** Replaced the `2>/dev/null || echo "[]"` pattern with explicit exit-code capture via `set +e` / `AZ_RC=$?` / `set -e`. The skip-the-guard branch now only triggers when `/tmp/az-err` contains `ResourceNotFound` or `was not found`; every other failure class (`AuthorizationFailed`, `AuthenticationFailed`, transient ARM 5xx, az CLI crash, network blip, malformed query, RG missing) now fails closed with the original stderr surfaced via `cat /tmp/az-err >&2`. This narrows the first-apply heuristic to genuine "container does not exist" and prevents silent CI fall-through on any other error.

### WR-01: Stale `jobrag-prod-aca` container name in README out-of-band-seeding runbook

Verified via repo-wide grep that no other operational typos exist — remaining `jobrag-prod-aca` references are either the legitimate Container App Environment name (`jobrag-prod-aca-env`, a different resource) or historical references in planning docs that document the typo trap.

### WR-02: `scripts/refresh-swa-origin.sh` mutates `prod.tfvars` BEFORE checking `prod.tfvars.local` exists

**Files modified:** `scripts/refresh-swa-origin.sh`
**Commit:** `ca3be1d`
**Applied fix:** Moved the `if [ ! -f prod.tfvars.local ]` guard ABOVE the `sed -i.bak` rewrite of `prod.tfvars`. The script now fails fast with exit code 2 BEFORE any in-place mutation occurs, so a missing `prod.tfvars.local` no longer leaves `prod.tfvars` half-rewritten with a stale SWA origin and no apply to settle it. The `.bak` cleanup was preserved on the success path. `bash -n` syntax check passed.

### WR-03: cpu:memory 1:2 ratio is documented but not validated — invalid Azure Consumption combos pass `terraform plan`

**Files modified:** `infra/modules/compute/main.tf`
**Commit:** `3ca3c1d`
**Applied fix:** Added a `precondition` block inside the existing `lifecycle` block on `azurerm_container_app.api`. The condition enumerates all 8 valid `(cpu, memory)` pairs of the Azure Consumption profile's 1:2 ratio (0.25/0.5Gi → 2.0/4Gi). Invalid combinations (e.g., `cpu=1.0, memory="0.5Gi"`) now fail at `terraform plan` time with an actionable error message listing all valid pairs, rather than blowing up at apply time with an opaque `ContainerAppInvalid` 4xx from ARM. `terraform fmt -check` and `terraform validate` both pass clean (one pre-existing unrelated deprecation warning in monitoring module).

### WR-04: CI guard's `.value // empty` silently treats secretRef env entries as empty

**Files modified:** `.github/workflows/deploy-infra.yml`
**Commit:** `f48c45a` (combined with CR-01 — same file, same shell block, same atomic logical change to the pre-apply guard)
**Applied fix:** Updated the jq filter from `.value // empty` to `(.value // .secretRef // empty)` so the guard accepts both env entry shapes — plain `.value` literal (current shape for the three auth vars) and `.secretRef` reference (if any of these vars ever get promoted to KV-managed). The fix is forward-compatible: today's plain-value entries still resolve correctly, and a future refactor that moves any of these to a `secretRef` won't silently mis-detect them as empty. Added an inline comment cross-referencing WR-04.
