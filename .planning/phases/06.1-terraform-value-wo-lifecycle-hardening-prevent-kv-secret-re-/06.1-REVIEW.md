# Phase 06.1: Code Review Report

**Reviewed:** 2026-05-26T10:00:00Z
**Depth:** standard
**Files Reviewed:** 9

## Summary

Phase 06.1 ships three orthogonal infra-hardening changes (KV `value_wo` lifecycle widening, ACA `cpu`/`memory` parameterization, dual-var-file invocation discipline) plus a CI pre-apply guard. The Terraform changes are clean and the wrapper script is well-structured. The README is thorough.

One **critical** issue: the CI pre-apply guard in `deploy-infra.yml` fails OPEN on any `az containerapp show` error (not just "container not found"), which defeats its entire defense-in-depth purpose. This is the same defect class as the typo already caught in-flight (`jobrag-prod-aca` → `jobrag-prod-api`, fix `cdf3ba3`): a guard that silently no-ops when its prerequisite read fails. Worth fixing in a sibling commit before Phase 06.1 closes.

Four **warnings**: a stale `jobrag-prod-aca` typo survives in the README runbook prose (same class as the CI typo), the cpu:memory 1:2 ratio is only documented in description text — not enforced at plan time even though independent `contains()` validation lists already exist, `scripts/refresh-swa-origin.sh` rewrites `prod.tfvars` BEFORE checking for `prod.tfvars.local` (leaves a half-applied state on failure), and the CI guard's `.value` jq filter silently misclassifies secretRef env entries as empty.

Two **info** items on documentation polish.

## Critical Issues

### CR-01: CI pre-apply guard fails OPEN on every `az containerapp show` error class except first-apply

**File:** `.github/workflows/deploy-infra.yml:79-91`

**Issue:** The guard reads the live ACA container template via:

```bash
ENV_JSON=$(az containerapp show \
  --name jobrag-prod-api \
  --resource-group jobrag-prod-rg \
  --query 'properties.template.containers[0].env' \
  -o json 2>/dev/null || echo "[]")

if [ "$ENV_JSON" = "[]" ]; then
  echo "::notice::Container app jobrag-prod-api not found — skipping pre-apply guard (first-apply path)."
  exit 0
fi
```

The `2>/dev/null || echo "[]"` swallows EVERY error class — `ResourceNotFound`, `AuthorizationFailed`, `AuthenticationFailed`, network blip, RG missing, az CLI crash, malformed query, transient ARM 5xx — and converts all of them into the "first-apply, skip the guard" branch. Any one of these fails CI OPEN with `::notice::skipping` and proceeds to `terraform apply -var-file=prod.tfvars`, which is the exact failure mode Bug #3 + this entire D-10 plan exists to prevent.

This is the same defect class as the in-flight typo Plan 04 caught (commit `cdf3ba3` — wrong container name → guard always skipped). The typo fix swapped the literal name but did not harden the error-handling shape, so a permissions blip, OIDC scope change, or temporary az CLI bug now re-opens the same hole.

The first-apply branch ALSO matches the literal JSON string `"[]"` — but a real container with no env vars (impossible here, but technically valid Azure state) would ALSO match `"[]"`. So even the happy-path discriminator is loose.

**Fix:**

```bash
# Capture stdout AND stderr separately; check az exit code explicitly.
set +e
ENV_JSON=$(az containerapp show \
  --name jobrag-prod-api \
  --resource-group jobrag-prod-rg \
  --query 'properties.template.containers[0].env' \
  -o json 2>/tmp/az-err)
AZ_RC=$?
set -e

if [ $AZ_RC -ne 0 ]; then
  if grep -q "ResourceNotFound\|was not found" /tmp/az-err; then
    echo "::notice::Container app jobrag-prod-api not found — skipping pre-apply guard (first-apply path)."
    exit 0
  fi
  echo "::error::az containerapp show failed (rc=$AZ_RC). Refusing to apply on unknown live state." >&2
  cat /tmp/az-err >&2
  exit 1
fi

# ... existing for loop ...
```

This narrows the skip-the-guard branch to the genuine "container does not exist" case and fails closed on every other error.

## Warnings

### WR-01: Stale `jobrag-prod-aca` container name in README out-of-band-seeding runbook

**File:** `infra/envs/prod/README.md:108-109`

**Issue:** The out-of-band-secret-seeding section instructs the operator to restart the container revision via:

```bash
az containerapp revision restart --name jobrag-prod-aca --resource-group "$RG" \
  --revision "$(az containerapp revision list --name jobrag-prod-aca --resource-group "$RG" --query '[0].name' -o tsv)"
```

The actual container app name is `jobrag-prod-api` (env-first), per `infra/modules/compute/main.tf:23` (`name = "jobrag-${var.env}-api"`). README line 266 already documents that the env-first naming has bitten before. The CI guard typo (commit `cdf3ba3`) was the exact same defect class — but it survives here in the runbook the operator follows by hand. Anyone copy-pasting these two lines after seeding a real secret will hit `ResourceNotFound` and may assume the seed itself failed.

**Fix:**

```bash
az containerapp revision restart --name jobrag-prod-api --resource-group "$RG" \
  --revision "$(az containerapp revision list --name jobrag-prod-api --resource-group "$RG" --query '[0].name' -o tsv)"
```

Sweep the rest of the README for the same string: `grep -n "jobrag-prod-aca" infra/envs/prod/README.md` — line 108 + line 109 are the two occurrences. The reference at line 266 is correct (it explicitly documents the typo trap and uses `jobrag-prod-api`).

### WR-02: `scripts/refresh-swa-origin.sh` mutates `prod.tfvars` BEFORE checking `prod.tfvars.local` exists

**File:** `scripts/refresh-swa-origin.sh:27-39`

**Issue:** The script's flow is:

1. Read SWA host (line 19) — fails fast if empty.
2. Rewrite `prod.tfvars` in place via `sed -i.bak` (lines 27-32).
3. Check that `prod.tfvars.local` exists (lines 36-39), exit 2 with FATAL message if missing.
4. Run `terraform apply` (line 41).

If `prod.tfvars.local` is missing (which is the failure mode the new guard exists to prevent), the script has ALREADY rewritten `prod.tfvars` before bailing. The `.bak` cleanup at line 29 removes the only easy rollback path. The operator is left with `prod.tfvars` mutated to point at a possibly-stale SWA origin and no apply to settle it. The next dual-var-file apply (whether via the wrapper or directly) will then propagate that mutation.

**Fix:** Move the `prod.tfvars.local` existence check ABOVE the sed rewrite:

```bash
# Phase 06.1 D-07 — pass 2 (CORS refresh) MUST also load prod.tfvars.local, or it
# silently re-wipes BACKEND_AUDIENCE / ENTRA_TENANT_* env vars (Bug #3). Guard up
# front so we never leave prod.tfvars half-rewritten.
if [ ! -f prod.tfvars.local ]; then
  echo "FATAL: prod.tfvars.local missing — see infra/envs/prod/README.md 'Apply command convention'" >&2
  exit 2
fi

# Re-write prod.tfvars in place
if grep -q '^swa_origin' prod.tfvars; then
  sed -i.bak "s|^swa_origin.*|swa_origin = \"$SWA_ORIGIN\"|" prod.tfvars
  rm -f prod.tfvars.bak
else
  echo "swa_origin = \"$SWA_ORIGIN\"" >> prod.tfvars
fi

terraform apply -var-file=prod.tfvars -var-file=prod.tfvars.local -auto-approve
```

### WR-03: cpu:memory 1:2 ratio is documented but not validated — invalid Azure Consumption combos pass `terraform plan`

**File:** `infra/modules/compute/variables.tf:109-127` (and `infra/envs/prod/variables.tf:162-180`)

**Issue:** Both `cpu` and `memory` variables carry independent `contains()` validation lists, but the cpu:memory 1:2 ratio constraint (called out in both descriptions and in the Plan 02 summary as the "Azure Consumption profile requirement") is enforced only by Azure at apply time, not at plan time. A user writing `cpu = 1.0, memory = "0.5Gi"` or `cpu = 0.25, memory = "4Gi"` will pass `terraform plan` cleanly, then eat an opaque `ContainerAppInvalid` 4xx from ARM on apply — exactly the kind of late-binding failure the per-variable `contains()` lists were added to prevent.

This is not a defect introduced by Phase 06.1 in the safety direction (the prior hardcoded 0.5/1Gi was always valid), but the new variable surface invites the foot-gun. Since the validation lists are already present per variable, the cross-variable rule is a natural follow-on.

**Fix:** Add a cross-variable validation either via a `validation` block on one variable that references the other (Terraform 1.9+ supports this), or via a `precondition` on the `azurerm_container_app.api` resource:

```hcl
# In infra/modules/compute/main.tf, inside the resource block:
lifecycle {
  ignore_changes = [
    template[0].container[0].image,
    template[0].revision_suffix,
  ]
  precondition {
    # Azure Consumption profile requires cpu:memory 1:2 ratio.
    # cpu=0.25→0.5Gi, 0.5→1Gi, 0.75→1.5Gi, ..., 2.0→4Gi.
    condition = (
      (var.cpu == 0.25 && var.memory == "0.5Gi") ||
      (var.cpu == 0.5  && var.memory == "1Gi")   ||
      (var.cpu == 0.75 && var.memory == "1.5Gi") ||
      (var.cpu == 1.0  && var.memory == "2Gi")   ||
      (var.cpu == 1.25 && var.memory == "2.5Gi") ||
      (var.cpu == 1.5  && var.memory == "3Gi")   ||
      (var.cpu == 1.75 && var.memory == "3.5Gi") ||
      (var.cpu == 2.0  && var.memory == "4Gi")
    )
    error_message = "Azure Consumption profile requires cpu:memory 1:2 ratio. Got cpu=${var.cpu}, memory=${var.memory}."
  }
}
```

Lower-effort alternative: leave validation as-is but add a comment block to `prod.tfvars` calling out the ratio explicitly so anyone tuning the values has the rule in front of them at the edit site.

### WR-04: CI guard's `.value // empty` silently treats secretRef env entries as empty

**File:** `.github/workflows/deploy-infra.yml:94`

**Issue:** The guard extracts each env var's value via:

```bash
VAL=$(echo "$ENV_JSON" | jq -r ".[] | select(.name==\"$VAR\") | .value // empty")
```

ACA env entries are a union type — they have EITHER `value` (plain literal) OR `secretRef` (KV/literal-secret reference), never both. The three vars the guard protects (`BACKEND_AUDIENCE`, `ENTRA_TENANT_ID`, `ENTRA_TENANT_SUBDOMAIN`) are plain literals per `infra/modules/compute/main.tf:173-184`, so the current jq filter works today. But if a future refactor ever moves any of these to a `secretRef` (e.g., promoting `BACKEND_AUDIENCE` to KV-managed), the guard will silently treat the env entry as "empty value" and fail CI closed with a misleading `Live ACA $VAR is empty` error — the operator's first instinct will be to re-seed prod.tfvars.local, which won't help.

**Fix:** Make the jq filter explicit about both shapes:

```bash
VAL=$(echo "$ENV_JSON" | jq -r ".[] | select(.name==\"$VAR\") | (.value // .secretRef // empty)")
```

This treats a populated `secretRef` as "present" (since the actual secret content isn't readable here anyway — we're just checking the env entry exists with non-empty wiring). A separate Action could then check secret resolution depth if needed. At minimum, the comment block (lines 52-75) should call out that the guard assumes plain-value shape; if anyone moves the vars to secretRef, the guard needs revisiting.

## Info

### IN-01: README "fails loudly" wording understates what the wrapper does

**File:** `infra/envs/prod/README.md:22`

**Issue:** The Apply command convention says the wrapper "fails loudly if `prod.tfvars.local` is missing." It actually exits with a specific exit code 2 and an 8-line remediation message to stderr (per `scripts/tf-apply-prod.sh:49-63`), plus a separate exit code 3 + usage for "no subcommand." Documenting the exit codes inline gives an operator running the wrapper under `set -e` in a parent script enough info to branch correctly.

**Fix:** Append to the wrapper paragraph:

> The wrapper exits 2 with a full remediation message if `prod.tfvars.local` is missing, 3 if no subcommand is supplied, and otherwise passes through Terraform's exit code (including `-detailed-exitcode`'s 0/1/2 semantics).

### IN-02: `prod.tfvars` `swa_origin` is already populated — first-apply path is stale

**File:** `infra/envs/prod/prod.tfvars:38`

**Issue:** The comment on line 36-37 says "First apply: leave empty → ALLOWED_ORIGINS = 'http://localhost:5173'. Second apply (after scripts/refresh-swa-origin.sh): script rewrites to 'https://<swa-default>'", but line 38 already has `swa_origin = "https://witty-flower-065dac003.7.azurestaticapps.net"` committed. Anyone bootstrapping a fresh prod environment from this repo will skip the two-pass CORS bootstrap (the wrapper will set ALLOWED_ORIGINS straight to the prior SWA's origin, which won't match the new SWA's hostname). This is pre-existing (not introduced by Phase 06.1) but the README's W2 ordered runbook still references the empty-then-refresh dance.

**Fix:** Out of scope for Phase 06.1, but worth flagging as a doc-vs-state drift for the next person who rebuilds prod. Either restore `swa_origin = ""` (forcing the runbook back to its documented shape) or update the README to say "if rebuilding from clean state, set `swa_origin = ""` first." Recommend a tiny follow-up commit but not blocking for 06.1.
