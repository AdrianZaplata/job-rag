# Phase 06.1: Terraform value_wo lifecycle hardening + ACA size parameterization + tfvars discipline — Context

**Gathered:** 2026-05-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 06.1 hardens `infra/envs/prod/` and `infra/modules/compute/` so that the next bare `terraform apply` against prod CANNOT reproduce the three UAT-blocking infra regressions discovered during Phase 06 UAT (M1 chat hang). Specifically:

1. **value_wo placeholder re-write** — 4 KV secret resources currently use `value_wo = "managed-out-of-band"` (or `var.seeded_user_entra_oid`) + `value_wo_version = 1`. Out-of-band `az keyvault secret set` calls seed the real OpenAI / Langfuse / seeded-user-oid values, but the `value_wo_version=1` trigger unconditionally writes the placeholder on the next apply, wiping the real values. `lifecycle.ignore_changes = [value]` (currently present) does NOT protect `value_wo`. This is Bug #1 + Bug #2 in 06-UAT-DEBUG-HANDOFF.md and the canonical incident behind `terraform-value-wo-overwrites-secret` memory.

2. **Hardcoded ACA cpu/memory** — `infra/modules/compute/main.tf:104-105` hardcodes `cpu = 0.5, memory = "1Gi"`. Adrian bumped the live revision to `cpu=1.0, memory=2Gi` via `az containerapp update` after OOM-killing the agent during Phase 06 UAT M1 (Bug #4). The next `terraform apply` would revert it, breaking chat again. The module must surface `cpu` and `memory` as variables (Azure Consumption profile constraint: cpu:memory ratio must be 1:2 — e.g., 0.5/1Gi, 0.75/1.5Gi, 1.0/2Gi).

3. **`prod.tfvars.local` invocation discipline** — `infra/envs/prod/prod.tfvars.local` carries `backend_audience`, `entra_tenant_id`, `entra_tenant_subdomain`, `home_ip`, `api_audience` — all required for the ACA container template to expose JWT auth env vars correctly. The committed runbook (`infra/envs/prod/README.md`) tells operators to run `terraform apply -var-file=prod.tfvars` (the committed file only) — which silently produces blank `BACKEND_AUDIENCE` / `ENTRA_TENANT_*` env vars on the container, breaking Entra auth (Bug #3). Adrian had to recover via `az containerapp update --set-env-vars` after the fact.

**Out of scope for 06.1:**
- Rewriting the out-of-band seeding model itself (Option B stays — secrets stay out of GitHub Actions secrets).
- Migrating any other module's KV secrets (the postgres-admin-password lives in the database module via `random_password` and is not affected by the value_wo trigger).
- Refactoring the two-pass CORS bootstrap (DEPL-12) or any other Phase 3 invariant.
- Backend code changes — this is a pure Terraform + runbook hardening phase.
- Reverting / re-architecting the value_wo migration itself (Phase 03-08 added it for D-13 full-fidelity; we keep value_wo, we just stop it from rewriting on subsequent applies).
- Live `terraform apply` against prod from CI — verification is a `terraform plan -detailed-exitcode` no-op proof, executed locally by Adrian. The actual remediation apply against the live tenant is left to Adrian's manual run after Phase 06.1 commits land.

<decisions>
## Implementation Decisions

### D-01 — value_wo lifecycle protection: add `value_wo_version` (and `value_wo`) to `ignore_changes`
For each of the 4 `azurerm_key_vault_secret` resources in `infra/envs/prod/main.tf` (openai_api_key, langfuse_public_key, langfuse_secret_key, seeded_user_entra_oid), the `lifecycle` block MUST be updated so subsequent applies cannot trigger a re-write of the seeded value:

```hcl
lifecycle {
  ignore_changes = [value, value_wo, value_wo_version]
}
```

**Why all three:** `value` is the legacy attribute (pre-AzureRM 4.x). `value_wo` is the write-only replacement. `value_wo_version` is the trigger — bumping it (or reverting it after an out-of-band seed) is what tells AzureRM to push `value_wo` again. Ignoring all three is the only way to make out-of-band-seeded secrets survive every future apply, regardless of which attribute drifts.

### D-02 — Apply D-01 to all 4 KV secret resources, including `seeded_user_entra_oid`
The `seeded_user_entra_oid` resource (`infra/envs/prod/main.tf:177-184`) currently has NO `lifecycle` block at all and sources `value_wo` from `var.seeded_user_entra_oid`. Add the same 3-attribute `ignore_changes` block. The variable still flows on first apply (per Phase 3 D-09 bootstrap); subsequent applies must not clobber any out-of-band rotation Adrian performs via `az keyvault secret set`.

### D-03 — Parameterize ACA `cpu` and `memory` as compute module variables
Add two new variables to `infra/modules/compute/variables.tf`:

```hcl
variable "cpu" {
  type        = number
  description = "Container vCPU allocation. Azure Consumption profile requires cpu:memory ratio of 1:2 (e.g., 0.25/0.5Gi, 0.5/1Gi, 0.75/1.5Gi, 1.0/2Gi). Bumped from 0.5 to 1.0 during Phase 06 UAT M1 (06-UAT-DEBUG-HANDOFF Bug #4) — agent OOM-killed at 1Gi while preloading the cross-encoder + serving an astream_events request."
  default     = 1.0
  validation {
    condition     = contains([0.25, 0.5, 0.75, 1.0, 1.25, 1.5, 1.75, 2.0], var.cpu)
    error_message = "cpu must be one of: 0.25, 0.5, 0.75, 1.0, 1.25, 1.5, 1.75, 2.0 (Azure Consumption profile)."
  }
}

variable "memory" {
  type        = string
  description = "Container memory allocation, must match cpu at 1:2 ratio. E.g., '2Gi' for cpu=1.0. See cpu variable description."
  default     = "2Gi"
  validation {
    condition     = contains(["0.5Gi", "1Gi", "1.5Gi", "2Gi", "2.5Gi", "3Gi", "3.5Gi", "4Gi"], var.memory)
    error_message = "memory must be one of: 0.5Gi, 1Gi, 1.5Gi, 2Gi, 2.5Gi, 3Gi, 3.5Gi, 4Gi (Azure Consumption profile)."
  }
}
```

Replace the hardcoded literals at `infra/modules/compute/main.tf:104-105` with `var.cpu` and `var.memory`. The defaults (1.0 / 2Gi) match the live revision Adrian bumped to during UAT, so a bare `terraform apply` with no var override produces NO container size diff.

### D-04 — Wire `cpu`/`memory` through the prod env composition (with explicit override)
In `infra/envs/prod/main.tf` `module "compute"` block, add `cpu = var.cpu` and `memory = var.memory`. Add matching `cpu` / `memory` variables to `infra/envs/prod/variables.tf` (re-using the same descriptions and validations as the module). Set `cpu = 1.0` and `memory = "2Gi"` in `infra/envs/prod/prod.tfvars` (committed) so the production size is version-controlled and visible in PRs, not buried in a local file. **Dev env intentionally untouched** — `infra/envs/dev/` is scaffold-only per Phase 3 D-04.

### D-05 — Single canonical tfvars invocation: BOTH files, every time
Adopt a uniform apply command for prod: `terraform apply -var-file=prod.tfvars -var-file=prod.tfvars.local`. Both files must be passed on every apply. Rationale:

- `prod.tfvars` (committed) carries non-secret, version-controlled config (image_tag, swa_origin, ghcr_username, cpu, memory, etc.).
- `prod.tfvars.local` (gitignored) carries Adrian-specific or per-machine values that should not be committed: `home_ip` (rotates with ISP DHCP), `backend_audience` / `entra_tenant_id` / `entra_tenant_subdomain` (real Entra GUIDs — committable in principle but kept local per established convention), `api_audience`.
- Passing only `prod.tfvars` (today's default per the README runbook) makes `backend_audience` / `entra_tenant_id` / `entra_tenant_subdomain` resolve to their `""` defaults in `infra/modules/compute/variables.tf`, which wipes the ACA env vars and breaks Entra auth (06-UAT-DEBUG-HANDOFF Bug #3).

**Rejected alternative — move values to `prod.tfvars`:** The user-facing option was "or move those values to a non-gitignored file." That is rejected because (a) `home_ip` is volatile, (b) `*.tfvars.local` is the established Adrian-machine convention per Phase 3 ghcr_pat handling, and (c) the Entra tenant subdomain `jobrag` is the marketing-visible CIAM domain — fine to commit but lower friction to keep the file pattern uniform. **D-05 makes the apply command the source of truth.**

### D-06 — Update `infra/envs/prod/README.md` runbook to use the dual-var-file invocation
Every `terraform apply` and `terraform plan` example in `infra/envs/prod/README.md` MUST be updated to include `-var-file=prod.tfvars.local` after `-var-file=prod.tfvars`. Affected sections (verified via `grep -n "var-file" infra/envs/prod/README.md`):
- Line 19 (Ordered runbook step 2 — pass 1 apply)
- Line 33 (First apply pass 1 code block)
- Line 85 (Two-pass CORS bootstrap)
- Line 107 (Image push section's first-apply hint)
- Line 198 (home_ip refresh procedure cross-reference — verify the database module README does the same)
- Line 202 (Drift detection plan command)
- Line 189 (ghcr_pat rotation runbook — `terraform apply -var ghcr_pat=...` should become `terraform apply -var-file=prod.tfvars -var-file=prod.tfvars.local -var ghcr_pat=...`)

Add a new "Apply command convention" callout near the top of the runbook (above the Ordered runbook section) that:
- Names the canonical command verbatim: `terraform apply -var-file=prod.tfvars -var-file=prod.tfvars.local`
- Explains why both files are required (D-05 rationale, 1–2 sentences)
- Provides a recommended shell alias / function or a `scripts/tf-apply-prod.sh` thin wrapper (see D-07).

### D-07 — Add a `scripts/tf-apply-prod.sh` thin wrapper as the recommended path
Create `scripts/tf-apply-prod.sh` that:
- Hard-codes the dual var-file invocation.
- Verifies `prod.tfvars.local` exists in `infra/envs/prod/` and exits non-zero with a clear remediation message if missing (e.g., "Missing infra/envs/prod/prod.tfvars.local — see infra/envs/prod/README.md 'Apply command convention'").
- Accepts pass-through args (`plan`, `apply`, `destroy`, `-auto-approve`, `-target=...`, etc.).
- Runs `cd infra/envs/prod` internally so it can be invoked from repo root.
- Uses `set -euo pipefail`.
- Is executable (`chmod +x`).

Update `scripts/refresh-swa-origin.sh` (the existing two-pass CORS script) to also pass `-var-file=prod.tfvars.local` on its internal `terraform apply` call — otherwise pass 2 (CORS refresh) silently re-wipes the Entra env vars even if pass 1 was correct.

### D-08 — Pre-flight verification: `terraform plan` must be a no-op after Phase 06.1 ships
Plan 4 (verification plan) MUST run `terraform plan -var-file=prod.tfvars -var-file=prod.tfvars.local -detailed-exitcode` from `infra/envs/prod/` against the live state and confirm exit code `0` (no changes). The plan author records the actual exit code + the `Plan: 0 to add, 0 to change, 0 to destroy.` line in `06.1-VERIFICATION-EVIDENCE.md`.

If the plan shows ANY container-size change (cpu/memory diff), D-03/D-04 defaults are wrong — re-tune the defaults until the live revision matches. If it shows ANY KV secret value/value_wo diff, D-01/D-02 ignore_changes is incomplete. If it shows any auth env-var diff (BACKEND_AUDIENCE etc.), prod.tfvars.local wasn't picked up.

**This is autonomous: false** — it requires Adrian's local az login against the prod subscription.

### D-09 — `prod.tfvars` becomes the source of truth for cpu/memory; document the file pyramid
Add an inline header comment to `infra/envs/prod/prod.tfvars` listing what belongs where:
- `prod.tfvars` (committed, this file) — non-secret, deployment-stable config: image_tag, cpu, memory, ghcr_username, swa_origin, location, tags.
- `prod.tfvars.local` (gitignored, per-machine) — Adrian-specific or rotating values: home_ip, backend_audience, entra_tenant_*, api_audience.
- `TF_VAR_*` env vars (CI/local secret) — true secrets: ghcr_pat (passed by deploy-infra.yml as TF_VAR_ghcr_pat).
- Out-of-band via `az keyvault secret set` (never in TF) — OpenAI key, Langfuse keys, seeded_user_entra_oid post-rotation.

### D-10 — CI `deploy-infra.yml` must also pass `-var-file=prod.tfvars.local` (or surface the policy)
Audit `.github/workflows/deploy-infra.yml`: if it runs `terraform apply` against `infra/envs/prod/`, it MUST either pass both var files OR explicitly skip `prod.tfvars.local` (since it's gitignored and not present in the CI runner) and instead supply each `prod.tfvars.local`-only variable via `TF_VAR_*` env vars sourced from GitHub Actions secrets / variables. The plan author records which option the workflow uses and ensures the chosen path covers `backend_audience`, `entra_tenant_id`, `entra_tenant_subdomain` (and `home_ip` if the CI runner ever adds firewall rules — likely not, but verify).

**Likely outcome:** CI ALREADY uses `TF_VAR_*` env vars for these (this is the standard OIDC pattern), in which case Plan 3 documents that fact in the runbook + adds a CI-vs-local matrix table. If CI is silently broken (using only `prod.tfvars`), it gets fixed in Plan 3.
