<objective>
Fix 5 of 5 (Phase 04.1) â€” Refactor `infra/external/main.tf` to drop the
dedicated `azuread_application_identifier_uri.api` resource and set
`identifier_uris = ["api://${azuread_application.api.client_id}"]` inline on
the `azuread_application.api` resource. The inline attribute is the reliable
form per azuread provider ~> 3.x; the dedicated resource has a known failure
mode where Terraform reports success but the live `identifierUris` list
stays empty, causing AADSTS500011 sign-in errors.

**Why:** Phase 4 close-out Checkpoint 5 deviation #7 cost ~1h of diagnosis
(blank sign-in page â†’ DevTools console showed `AADSTS500011: The resource
principal named api://... was not found in the tenant`) before the
workaround (`az ad app update --identifier-uris â€¦`) bridged the gap. The
Terraform state was clean â€” only the live resource was empty. The reliable
form is the inline `identifier_uris` attribute on `azuread_application`,
which the provider treats as a managed attribute of the application (not as
a separate resource that can silently fail).

**Source of truth for this deviation:**
`.planning/phases/04-frontend-shell-auth/04-06-SUMMARY.md` deviation #7
("azuread_application_identifier_uri resource silently fails to populate")
+ memory `terraform-azuread-identifier-uri-unreliable.md`.

**Output:** Refactored `infra/external/main.tf` + operator note appended to
prod README "Phase 4 close" section + one atomic commit. **No
`terraform apply`** â€” the live Entra resource state reconciliation happens
out-of-band on the user's next `infra/external/` apply. The refactor is
verifiable via `terraform validate` + `terraform fmt -check` (both purely
local â€” no Azure API calls).
</objective>
