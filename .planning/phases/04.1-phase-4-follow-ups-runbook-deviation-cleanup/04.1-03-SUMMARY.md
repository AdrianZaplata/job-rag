# Phase 04.1 Plan 03: main.tsx boot-time try/catch + BootErrorFallback Summary

**`frontend/src/main.tsx` now wraps the top-level `await msalInstance.initialize()` / `await msalInstance.handleRedirectPromise()` / `createRoot().render()` chain in a try/catch — when the chain rejects (e.g. AADSTS500011 from the Terraform identifier_uri drift fixed in 04.1-05), a hermetic Tailwind-only `BootErrorFallback` surfaces the error + a Reload button instead of leaving the user on a blank `<div id="root">`.**

## Decisions Made

- **`rootEl!` non-null assertion inside renderBootError** (chosen over the plan's defensive `if (!rootEl) return` fallback). `frontend/eslint.config.js` extends only `tseslint.configs.recommended`, NOT `strict`, so `@typescript-eslint/no-non-null-assertion` is off. The assertion documents at the use-site that the null-check above the try guarantees rootEl is non-null. Single-line cleaner than re-checking.
- **`userEvent` (default import from `@testing-library/user-event`) for the click test** even though no other test in the suite uses userEvent (existing tests use `fireEvent`). The package is in devDependencies (^14.6.1); default import is canonical for v14; userEvent.setup() better models the async click event for the reload-button test. Trade-off: introduces a second event-firing pattern in the test suite — acceptable for the targeted new test file.
- **Hermetic fallback (no `ErrorBoundaryFallback` reuse)** per D-16. ErrorBoundaryFallback imports shadcn `Card`/`Button` + `react-router useNavigate` — exactly the subsystems that may have failed in the boot rejection path. Duplicating ~20 lines of layout markup is the correct trade-off; reusing would re-enter the failure surface.
- **Provider nesting preserved verbatim** inside the try block (MsalProvider → QueryClientProvider → BrowserRouter → ErrorBoundary → App). Plan task 02 was explicit about not changing nesting. The try/catch is a thin shell — zero drift to the live render path.
