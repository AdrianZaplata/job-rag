**SEEDED_USER_ENTRA_OID UPDATE migrated from Alembic 0005's upgrade() into init_db()::_seed_entra_oid() — rotation now requires only an ACA restart, not manual az containerapp exec.**

## Accomplishments
- Removed the once-per-revision UPDATE bug: rotating SEEDED_USER_ENTRA_OID is now an idempotent post-`alembic upgrade head` step that runs on every container boot.

## Decisions Made

- **Helper placement:** Kept `_seed_entra_oid()` in `engine.py` rather than spinning up a new `seed.py` next door. After the change, `engine.py` is ~75 LOC and stays single-purpose (sync/async engines + the boot hook that uses them); a separate file would split the helper from its only consumer with no readability gain. CONTEXT D-09 explicitly allowed either placement; executor's discretion picked the shorter path.
- **SEEDED_USER_UUID retained in migration:** Documents the migration's historical intent. Removing it would lose the labelled cross-reference any future operator reads when they grep for the UUID in `alembic/`. The constant is duplicated (not imported) in `engine.py` to keep migrations free of `job_rag.config` imports per Phase 1 D-08.
- **Defensive engine patch on existing init_db test:** The existing `test_init_db_invokes_alembic_upgrade` only mocked `command.upgrade`; under the rewritten `init_db` body it would have called the real `engine.begin()` if `SEEDED_USER_ENTRA_OID` happened to be set in the runner's env (rare, but possible for an operator who exports it locally). Wrapping the call site in an additional `patch("job_rag.db.engine.engine")` keeps the test deterministically SQL-free without changing its original assertion semantics.
