## Warnings

### WR-01: `_seed_entra_oid()` bypasses `settings` and reads `os.environ` directly

**File:** `src/job_rag/db/engine.py:76`
**Issue:** `_seed_entra_oid()` calls `os.environ.get("SEEDED_USER_ENTRA_OID", "").strip()` instead of reading `settings.seeded_user_entra_oid`. The same env var is already exposed on `Settings` (`src/job_rag/config.py:67`) and is consumed via `settings.seeded_user_entra_oid` by `src/job_rag/api/auth.py:146`. This creates two source-of-truth paths for a single config value:

- AUTH-06 (request path): `settings.seeded_user_entra_oid` — loaded once at process start, pydantic-validated, mockable in tests via `monkeypatch.setattr("job_rag.config.settings", ...)`.
- `_seed_entra_oid` (boot path): `os.environ.get(...)` — bypasses pydantic validation, requires `monkeypatch.setenv()` in tests, won't see `.env` file changes the way `Settings` does.

In practice this is mostly cosmetic (both paths read the same OS env on container boot), but it's exactly the drift surface CONTEXT.md and Pitfall 18 warn against, and the comment block on line 13-16 acknowledges the parallel concern for `SEEDED_USER_UUID`. If `Settings` ever adds field-level normalization (e.g. lowercasing, stripping curly braces from a GUID) on `seeded_user_entra_oid`, AUTH-06 and the boot bridge would silently disagree — same-shaped values from two reads is not the same as same value from one read.

**Fix:**
```python
from job_rag.config import settings  # already imported

def _seed_entra_oid() -> None:
    """..."""
    oid = settings.seeded_user_entra_oid.strip()
    if not oid:
        return
    with engine.begin() as conn:
        conn.execute(
            text(
                "UPDATE users SET entra_oid = :oid "
                "WHERE id = :seeded_uuid AND (entra_oid IS NULL OR entra_oid != :oid)"
            ).bindparams(oid=oid, seeded_uuid=SEEDED_USER_UUID)
        )
```

This also requires `tests/test_cli.py:78` to switch from `monkeypatch.setenv(...)` to `monkeypatch.setattr("job_rag.db.engine.settings.seeded_user_entra_oid", test_oid)` (or equivalent pydantic-aware mock).

### WR-02: `SEEDED_USER_UUID` redeclared as string literal in `engine.py` instead of imported from `settings`

**File:** `src/job_rag/db/engine.py:16`
**Issue:** The constant `SEEDED_USER_UUID = "00000000-0000-0000-0000-000000000001"` is a hardcoded duplicate of `settings.seeded_user_id` (defined in `src/job_rag/config.py:48` as `uuid.UUID("00000000-0000-0000-0000-000000000001")`). The comment block on line 13-15 explicitly names this as canonical (`"must match config.py settings.seeded_user_id"`), and `alembic/versions/0002_add_user_profile.py:25` already establishes the no-duplication pattern with:

```python
# Pin SEEDED_USER_ID to the Settings literal — prevents drift across the
# three places this UUID appears (config.py, this migration, tests). Do
# NOT re-declare the UUID literal here (Pitfall 18 regression surface).
SEEDED_USER_ID = settings.seeded_user_id
```

`engine.py` already imports `settings` (line 11), so the drift-free form is a one-line change. The justification for migration files duplicating constants (avoid full-app import at migration runtime) does NOT apply here — `engine.py` is part of the app and already pulls `settings`.

Pitfall 18 calls this out explicitly: every place this UUID is re-declared as a literal is a regression surface where future renames silently desync.

**Fix:**
```python
# Line 13-16 — replace the literal:
SEEDED_USER_UUID = str(settings.seeded_user_id)
```

(Cast to `str` because the existing SQL bindparam treats it as a string; or change the bindparam to pass the `UUID` object directly and let asyncpg/psycopg2 handle it.)

### IN-02: `BootErrorFallback.test.tsx` `afterEach` cleanup uses pre-`defineProperty` reload reference

**File:** `frontend/src/test/BootErrorFallback.test.tsx:8-24`
**Issue:** The pattern:

```ts
let originalReload: () => void
beforeEach(() => {
  originalReload = window.location.reload  // captures the post-jsdom default
  Object.defineProperty(window, 'location', {
    configurable: true,
    value: { ...window.location, reload: vi.fn() },
  })
})
afterEach(() => {
  Object.defineProperty(window, 'location', {
    configurable: true,
    value: { ...window.location, reload: originalReload },
  })
})
```

works for THIS test file but is fragile cross-test: by the time `afterEach` runs, `window.location` is the spread-replaced object from `beforeEach`. `{...window.location}` shallow-copies that, then re-installs `originalReload`. Net effect is correct here but only because nothing else in the suite reads `window.location` between tests. If another vitest file in the same worker depends on the real (non-spied) `location` shape, this leaks a non-prototype-equal object that lacks `Location.prototype`'s methods.

The lower-risk modern jsdom idiom is `vi.stubGlobal('location', {...})` + `vi.unstubAllGlobals()` in `afterEach`, which restores the prototype-correct original. Not a bug for the standalone Phase 04.1 case.

**Fix:**
```ts
beforeEach(() => {
  vi.stubGlobal('location', { ...window.location, reload: vi.fn() })
})
afterEach(() => {
  vi.unstubAllGlobals()
})
```

### IN-03: Magic number `1000` for stack truncation in `BootErrorFallback`

### IN-04: `test_init_db_invokes_alembic_upgrade` `pytest.skip` paths are now unreachable post-Plan-02

**File:** `tests/test_cli.py:24-44`
**Issue:** The test still carries three `pytest.skip()` branches that gate on Plan 02 not having landed yet:

- `except ImportError: pytest.skip("CLI import failed - confirm job_rag.cli exports Typer app")` (line 27)
- `except (AttributeError, ModuleNotFoundError): pytest.skip("alembic.command not yet imported in db/engine.py (Plan 02)")` (line 42)
- `if mock_upgrade.call_count == 0: pytest.skip("init_db not yet swapped to alembic command (Plan 02 provides it)")` (line 44)

Plan 02 has long since landed (the import block at the top of `src/job_rag/db/engine.py` shows `from alembic import command`, and `init_db()` calls `command.upgrade(cfg, "head")`). These three skip paths now hide real regressions — if anyone breaks the alembic import or removes `init_db`'s alembic call, the test will skip green instead of failing red.

Not strictly a Phase 04.1 introduction (these dates back to Plan 02 transition), but the file was edited in this phase to add `_seed_entra_oid` mocking, making it the right cleanup window.

**Fix:** Replace each `pytest.skip(...)` with `pytest.fail(...)` or `raise`, e.g.:

```python
try:
    from job_rag.cli import app as cli_app
except ImportError as e:
    pytest.fail(f"CLI import failed - job_rag.cli must export Typer app: {e}")
```

The two new `_seed_entra_oid` tests below (lines 51-96) already use the no-skip modern form and are good models.
