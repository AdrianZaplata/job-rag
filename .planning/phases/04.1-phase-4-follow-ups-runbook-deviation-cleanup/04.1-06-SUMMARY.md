## Patterns Established (and Lessons Learned)

Three patterns crystallized over the course of executor + operator work:

1. **The split pattern is THE structural fix.** `azuread_application_registration` (base, with `requested_access_token_version = 2` as a direct top-level attribute) + `azuread_application_identifier_uri` (sibling, cross-references registration.client_id — plans cleanly) + `azuread_application_permission_scope` (sibling, carries the OAuth2 scope) is the canonical decomposition when an `api://${self.client_id}` URI shape is required. `azuread_application_api_access` is the WRONG abstraction (it's for OUTBOUND scope grants; requires `api_client_id`). Backstopped by the script-level `az ad app show --query identifierUris` live-state assertion in `refresh-external-outputs.sh`.

2. **Schema introspection IS planning** (process guardrail). Phase 04.1-05 conflated 'validate passes' with 'plan passes'; revision-0 of Phase 04.1-06 conflated 'resource exists' with 'resource carries this attribute'. The fix is the same one level up: `terraform init -backend=false && terraform providers schema -json | jq` against the locked provider version BEFORE asserting any attribute. Task 01 step 1 institutionalized this; permanent guardrail for any future Terraform refactor in this repo.

3. **NEW from operator action (opposite of what the plan + memory predicted):** Terraform's `azuread_service_principal_delegated_permission_grant` propagates **AllPrincipals admin consent automatically** when re-created in the same apply as the SP replacement. The plan's "must re-grant via portal" warning was overcautious — the operator never touched the portal. `az ad app permission list-grants` returned `consentType=AllPrincipals` immediately post-apply. Capture for future SP-replace operations: if the delegated-permission-grant resource is in the same plan as the SP being replaced, TF handles consent migration end-to-end.

## Unplanned Deviation (Rule 1) — Disclosure

**Commit `c28bb13` was NOT in the plan but was REQUIRED to make sign-in work.**

After Task 09 step 7 (`gh secret set VITE_API_AUDIENCE`), the operator initially hit AADSTS500011 on sign-in. Root cause: Vite inlines `VITE_*` env vars at BUILD time, so the SWA still served bundles built against the OLD `api_client_id` (`a12dfd07-...`). The existing `deploy-spa.yml` workflow only fired on `push` events with `paths: frontend/**` changes; a `gh secret set` doesn't touch `frontend/`, so no rebuild was triggered.

**Fix (atomic commit `c28bb13`):**
- Added `workflow_dispatch:` trigger to `.github/workflows/deploy-spa.yml` (enables manual SPA rebuild via `gh workflow run`).
- Rolled `frontend/.env.production VITE_API_AUDIENCE` from `api://a12dfd07-...` to `api://f4ced229-...` (ensures local-dev SPA build matches the new GUID; the inlining at build time also takes the GitHub secret value when building in CI).

Both changes were combined into one commit because they close the same loop on the secret rotation. Pushing master triggered Deploy SPA #8, which rebuilt the bundle from the new GitHub secret; after that deploy completed, sign-in worked.

**Pattern captured for future:** see `patterns-established` block above ("Vite build-time env vars + Static Web Apps deployment requires manual rebuild on secret rotation"). After every `gh secret set VITE_*`, follow with `gh workflow run deploy-spa.yml --ref master` to force a rebuild — OR roll a sentinel file under `frontend/` in the same commit. The `workflow_dispatch:` trigger landed here makes the manual-rebuild path permanently available.
