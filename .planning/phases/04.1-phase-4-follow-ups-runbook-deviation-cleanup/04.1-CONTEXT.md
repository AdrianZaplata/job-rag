**In scope (5 fixes, one PLAN per fix, executed in priority order Fix 1 → Fix 5):**

1. **Fix 1 (Structural — Medium):** Move `0005_adopt_entra_oid.py::upgrade()`'s `UPDATE users SET entra_oid = :oid …` into `init_db()` in `src/job_rag/db/engine.py` as an idempotent post-`alembic upgrade head` step. Rationale: Alembic `upgrade()` runs once per revision-marker; rotating `SEEDED_USER_ENTRA_OID` post-0005 currently requires manual `az containerapp exec` (see SUMMARY deviation #3 + memory `alembic-update-runs-once.md`).
2. **Fix 2 (CI cosmetic — Low):** Update the smoke-check poll in `.github/workflows/deploy-api.yml` so both `Running` AND `RunningAtMaxScale` are treated as healthy terminal states. Rationale: new revisions that scale-out during the 150s budget reach `RunningAtMaxScale` and are rejected by the literal `Running` check, producing red CI on successful deploys (see SUMMARY deviation #4 + memory `aca-deploy-verifier-trap.md`).
3. **Fix 3 (Fragility — Low):** Wrap the top-level `await msalInstance.initialize()` / `await msalInstance.handleRedirectPromise()` / `createRoot().render()` chain in `frontend/src/main.tsx` with `try { … } catch (e) { renderBootError(e) }` so React mounts and a styled error surface is shown when MSAL rejects (e.g. `AADSTS500011`). Rationale: top-level `await` rejection leaves a blank `<div id="root">` with no `ErrorBoundary` (because React never mounted) — see SUMMARY deviation #10.
4. **Fix 4 (Doc — Medium):** Document customer-vs-B2B-guest namespace + `az ad user create` customer/Member bootstrap step in `infra/bootstrap/README.md` (the Phase 3 tenant-creation runbook). `infra/envs/prod/README.md` already has the short-form note (line 223 from the Phase 4 close commit), but the *source of truth* for the tenant-creation runbook is `infra/bootstrap/README.md` and that's where future operators look first. Rationale: every re-bootstrap will hit deviation #6 until this is documented at the tenant-creation step (SUMMARY follow-up #4 + memory `ciam-customer-vs-b2b-guest.md`).
5. **Fix 5 (Terraform — Low):** Refactor `infra/external/main.tf` to drop the dedicated `azuread_application_identifier_uri.api` resource and set `identifier_uris = ["api://${...}"]` inline on the `azuread_application.api` resource. Rationale: the dedicated resource silently reports created in state while `az ad app show --query identifierUris` returns `[]` (deviation #7 + memory `terraform-azuread-identifier-uri-unreliable.md`). Inline attribute on `azuread_application` is the reliable form.

- Deviation #9 (CSP hardening in `staticwebapp.config.json`) — deferred to Phase 8 portfolio polish per SUMMARY (priority: Defer).

### Phase-wide constraints (user-locked)

- **D-01 (no remote pushes):** No `git push` from any plan. Every plan ends with a local atomic commit; the user pushes manually when they're ready to land all 5.
- **D-02 (no cloud touches):** No `az …`, no `terraform apply`, no `gh workflow run`, no `gh secret set`, no `az keyvault secret set`, no ACA restart, no Entra resource mutation. Verification steps may run `terraform validate` (purely local — `terraform init -backend=false` downloads providers but makes no API calls) and `terraform fmt -check`. Verification may run `pytest`, `uv run ruff check`, `uv run pyright`, `npm run build`, `npm run typecheck`, `npm run lint`, `vitest run`. CI workflow YAML files can be syntax-checked locally via `python -c "import yaml; yaml.safe_load(open(...))"` (PyYAML ships in the Python toolchain).
- **D-03 (one atomic commit per fix):** Each PLAN.md results in exactly one commit when run by `/gsd-execute-phase`. Each commit message follows the project convention (`fix(04.1-NN): <short description>` for code/CI, `docs(04.1-NN): <short description>` for doc-only fix 4). No combined commits, no fixup commits.
- **D-04 (priority order = serial execution):** Plans run in numeric order 04.1-01 → 04.1-05 via `depends_on` chain (each plan depends on the previous one). This guarantees the user's stated priority order. No parallel waves.
- **D-05 (canonical context):** Every plan's `<context>` block references `.planning/phases/04-frontend-shell-auth/04-06-SUMMARY.md` as the source of truth for the deviation it fixes. Executors reading the plan get the full deviation context (symptom, diagnosis, workaround, structural fix) without re-deriving it.
- **D-06 (no Phase 4 reopen):** Phase 4 is COMPLETE (verified by Adrian's live end-to-end sign-in 2026-05-21 17:30 UTC + 13/13 requirements closed). This phase does NOT re-verify Phase 4 requirements; the fixes are independent of the live state. No plans here add to or amend `REQUIREMENTS.md` AUTH-*/SHEL-* rows.

### Fix-specific decisions

#### Fix 1: init_db UPDATE refactor

- **D-07:** UPDATE moves from `alembic/versions/0005_adopt_entra_oid.py::upgrade()` into a new function `_seed_entra_oid()` called from `init_db()` AFTER `command.upgrade(cfg, "head")`. Same SQL (`UPDATE users SET entra_oid = :oid WHERE id = :seeded_uuid AND (entra_oid IS NULL OR entra_oid != :oid)`), same idempotency guard (skip on empty `SEEDED_USER_ENTRA_OID`), same canonical UUID constant.
- **D-08:** The UPDATE block stays REMOVED from `0005_adopt_entra_oid.py::upgrade()` body (do not leave a duplicate; new boots with a fresh DB still get the row from `0002_add_user_profile.py` and the UPDATE from `init_db()`). The `_has_column`/`_has_index` idempotency helpers and the schema bits (add `entra_oid` column if missing + partial unique index) stay in `0005`.
- **D-09:** `_seed_entra_oid()` uses the **sync `engine`** (already in `engine.py`) — `init_db()` is sync and runs at startup before the async engine is needed. Uses `engine.begin() as conn: conn.execute(sa.text(...).bindparams(oid=..., seeded_uuid=...))`.
- **D-10:** Test coverage: extend `tests/test_cli.py::test_init_db_invokes_alembic_upgrade` shape (already mocks `command.upgrade`) with two new cases: (a) `SEEDED_USER_ENTRA_OID` unset → no UPDATE executed, (b) `SEEDED_USER_ENTRA_OID="…"` → UPDATE executed with bind params. Mock the engine/connection (no live DB needed). No real Postgres required — keeps the test runnable without `docker compose up`.
- **D-11:** Migration 0005 keeps the comment block but the docstring updates to note "UPDATE moved to init_db() in 04.1-01 — see SUMMARY deviation #3".

#### Fix 2: deploy-api.yml smoke fix

- **D-12:** Change `if [ "$STATE" = "Running" ]; then` (line 151) to `if [ "$STATE" = "Running" ] || [ "$STATE" = "RunningAtMaxScale" ]; then`. Single-line edit; preserves the existing 150s poll budget and per-revision FQDN smoke. The iteration-log line ("Iteration $i: /health 200 but runningState='$STATE'") still fires for other transient states.
- **D-13:** Verification: `python -c "import yaml; yaml.safe_load(open('.github/workflows/deploy-api.yml'))"` (YAML syntax). Optional: `actionlint` if installed locally (not required — pre-existing CI step covers it).
- **D-14:** No test added — bash conditional in YAML is exercised by the next real deploy. Memory `aca-deploy-verifier-trap.md` gets a brief amendment line noting "Fix landed 04.1-02".

#### Fix 3: main.tsx try/catch

- **D-15:** New helper `renderBootError(error: unknown)` lives inline in `main.tsx` (small, single-use, doesn't justify a new file). It mounts a minimal styled fallback to `#root` using `createRoot(rootEl).render(<BootErrorFallback error={…} />)` so React DOES initialize and the user sees a readable surface instead of a blank `<div>`.
- **D-16:** `BootErrorFallback` is a tiny inline component (or function returning JSX) styled with raw Tailwind utility classes — does NOT import shadcn `Card`/`Button` to avoid risking the same eval-pathway / MSAL-init failure cascading into the fallback. Visual style: centered card, "We hit a problem starting the app." headline, monospace error message in a `<pre>`, "Reload" button calling `window.location.reload()`. No router, no MSAL — purely synchronous DOM.
- **D-17:** Wrap structure:
  ```ts
  try {
    await msalInstance.initialize()
    await msalInstance.handleRedirectPromise()
    createRoot(rootEl).render(<StrictMode>…</StrictMode>)
  } catch (error) {
    renderBootError(error)
  }
  ```
  The `rootEl` null-check stays OUTSIDE the try (it's a programmer error in `index.html`, not a runtime failure mode worth a styled surface).
- **D-18:** Test coverage: add a Vitest unit test for `BootErrorFallback` (renders error message in `<pre>`, reload button is present). End-to-end coverage of the try/catch path is out of scope (would need a Playwright/MSW MSAL-failure injection harness — too much for a one-commit fix). The visual smoke happens on the next prod deploy.
- **D-19:** Comment in `main.tsx` references SUMMARY deviation #10 inline so future readers understand why the wrap exists.

#### Fix 4: CIAM customer-bootstrap docs

- **D-20:** Primary target = `infra/bootstrap/README.md`. Add a new section (suggested title: "Bootstrap a first-login customer/Member account") with: (a) why B2B guests don't work (CIAM user flow disjointness — short paraphrase of memory `ciam-customer-vs-b2b-guest.md`), (b) the exact `az ad user create --user-principal-name adrian@<subdomain>.onmicrosoft.com --display-name … --password '…' --force-change-password-next-sign-in false` command against the External tenant, (c) where to capture the resulting OID for the KV `seeded-user-entra-oid` secret + `users.entra_oid` UPDATE, (d) the cross-reference to `04-06-RUNBOOK.md` Checkpoint 5 for the wider runbook.

Do NOT duplicate the full bootstrap snippet there.
- **D-22:** No code change. No infra apply. Pure documentation; verification is `grep` for the required anchor strings (e.g. `grep -q "az ad user create" infra/bootstrap/README.md`).
- **D-23:** Out of scope for this fix: configuring CIAM federation IdPs for B2B guests. SUMMARY follow-up #4 calls this as the alternative path; documenting customer-bootstrap is the simpler, lower-cost option for v1 (single user) and what the user has been doing in practice.

#### Fix 5: identifier_uri inline refactor

- **D-24:** Remove the entire `resource "azuread_application_identifier_uri" "api" { … }` block (lines 41-47 of `infra/external/main.tf`). Add `identifier_uris = ["api://${azuread_application.api.client_id}"]` as an attribute of the `azuread_application.api` resource. Per azuread provider ~> 3.0 docs, `identifier_uris` is a `Set of String` attribute supported directly on `azuread_application`. The original chicken-and-egg comment (lines 41-42) becomes moot — Terraform's two-pass apply (plan → create with client_id known → set identifier_uris) handles the dependency internally because the attribute references its own resource's `client_id` (which the provider computes after the create step).
- **D-25:** No state-file mutation in this phase. `infra/external/terraform.tfstate` will list `azuread_application_identifier_uri.api` as orphaned after the refactor lands; on the next apply (out-of-band, by the user, after this phase), Terraform plans a destroy of the dedicated resource + an update of `azuread_application.api` adding `identifier_uris`. That apply lives in the next operator session, NOT in this phase.
- **D-26:** Verification: `cd infra/external && terraform init -backend=false && terraform validate && terraform fmt -check`. `terraform validate` does NOT call Entra/Microsoft Graph — it only checks HCL syntax + schema. `init -backend=false` downloads the azuread provider plugin (~25MB) from the registry but makes no Azure-resource API calls. Acceptable per D-02 (provider registry is HashiCorp, not "cloud" for the purposes of the constraint).
- **D-27:** Operator notes: append to `infra/envs/prod/README.md` "Phase 4 close" section (or the runbook section, wherever it lives) one bullet: "Identifier URI refactor — on the next `infra/external/` apply, Terraform will plan a destroy of `azuread_application_identifier_uri.api` + an update on `azuread_application.api` adding `identifier_uris`. Approve both. The inline attribute is the reliable form (deviation #7 fix)."
- **D-28:** Memory `terraform-azuread-identifier-uri-unreliable.md` gets a brief addendum: "Structural fix landed 04.1-05 — moved identifier_uri inline on azuread_application resource. Re-test on next apply."
