<objective>
**Land a working `infra/external/` Terraform config that converges inline `identifier_uris` derivation via the `azuread_application_registration` + `azuread_application_identifier_uri` + `azuread_application_permission_scope` split pattern (with `requested_access_token_version = 2` set directly on the registration as a top-level attribute), with script-level `az ad app show` verification mitigating the documented silent-fail risk on `azuread_application_identifier_uri`, and atomically updates all dependent operator docs + memory.**

**Why:** Plan 04.1-05 shipped an invalid structural refactor — setting `identifier_uris = ["api://${azuread_application.api.client_id}"]` inline on the `azuread_application.api` block is a Terraform-graph-level self-reference (`terraform validate` passes the schema check, but `terraform plan` aborts with `Error: Self-referential block`). The provider has NO inline workaround for the `api://<self.client_id>` case. Reverting to the original standalone `azuread_application_identifier_uri.api` is unacceptable — it re-exposes the documented silent-fail mode (memory `terraform-azuread-identifier-uri-unreliable.md`). And the `api://<api_client_id>` URI shape is LOAD-BEARING per `src/job_rag/api/auth.py:72` (`removeprefix("api://")` extracts the client_id for JWT validation), so a static-string URI is NOT a valid alternative.

**Fix:** Split `azuread_application.api` into THREE provider-native sibling resources:
1. `azuread_application_registration.api` — lightweight base; carries `display_name`, `sign_in_audience`, AND `requested_access_token_version = 2` (this attribute IS supported directly on the registration in v3.8.0 — verified via `terraform providers schema -json` in Task 01 step 1).
2. `azuread_application_identifier_uri.api` — side resource referencing the registration's `client_id` (now a cross-resource reference, plans cleanly).
3. `azuread_application_permission_scope.access_as_user` — side resource carrying the OAuth2 scope (value, type, four consent description/display_name fields) that used to live inside the nested `api { oauth2_permission_scope { … } }` block.

**Critical correction from Issue 1 of the checker review:** `azuread_application_api_access` is NOT used in this refactor. That resource is for an app declaring its OUTBOUND access to ANOTHER api's scopes (the equivalent of the SPA's `required_resource_access` block) — it does NOT carry an app's OWN scope definitions, and it does NOT carry `requested_access_token_version`. The earlier draft of this plan (revision 0) mistakenly listed it; Task 01 now explicitly does NOT create it.

Mitigate the silent-fail risk on the URI side-resource by adding a post-`terraform output` `az ad app show` assertion to `scripts/refresh-external-outputs.sh` that aborts non-zero if the live `identifierUris` list is empty — fail-loud at the moment downstream `.env.production` / `prod.tfvars.local` / GitHub secrets would be rewritten with stale values.

**Approach lesson (added per Issue 4):** Before writing any HCL, Task 01 step 1 runs `terraform providers schema -json | jq` against the LOCKED provider version (`azuread = 3.8.0`) to verify which resources exist and which attributes they carry. The Phase 04.1-05 failure mode — "the plan author conflated 'validate passes' with 'plan passes'" — is mirrored at one level up if a planner asserts a resource carries an attribute without checking the schema. Schema introspection IS planning. This step is permanent process guardrail, not optional reconnaissance.

# Phase 04.1-06 — fail-loud guard against the documented silent-fail mode on
# azuread_application_identifier_uri (memory terraform-azuread-identifier-uri-unreliable.md).
# terraform output can return a stale or empty value if the underlying resource
# applied "successfully" but the live identifierUris list stayed empty. We assert
# live state directly via Microsoft Graph BEFORE rewriting any downstream files.
if [ -n "$API_CLIENT_ID" ]; then
  echo "Verifying live identifierUris for API app $API_CLIENT_ID ..."
  LIVE_URIS="$(az ad app show --id "$API_CLIENT_ID" --query 'identifierUris' -o tsv 2>/dev/null || true)"
  if [ -z "$LIVE_URIS" ]; then
    echo "FATAL: az ad app show returned empty identifierUris for $API_CLIENT_ID" >&2
