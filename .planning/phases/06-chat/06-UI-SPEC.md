## 1. Inherited Design System (one-line affirmation)

Phase 6 uses the exact design system Phase 4 shipped and Phase 5 affirmed:

- **shadcn preset (on disk):** `style=radix-nova`, `baseColor=neutral`, `iconLibrary=lucide`, `cssVariables=true`, `tsx=true`, `aliases={ ui: "@/components/ui", utils: "@/lib/utils" }`. **Phase 6 does NOT re-init shadcn or pass `--style` / `--base-color` flags during `add` commands.**
- **Tailwind:** v4 via `@tailwindcss/vite`; theme tokens declared in `frontend/src/app.css` `@theme inline` block.
- **Font:** Geist Variable (`@import "@fontsource-variable/geist"` in `app.css`). No mono font family is installed; mono is achieved via `font-mono` Tailwind utility which falls back to the system mono stack (`ui-monospace, SFMono-Regular, Menlo, Monaco, "Cascadia Mono", "Segoe UI Mono", "Roboto Mono", monospace`). This is load-bearing in Phase 6 for tool names, JSON args, and JSON output (consistent with Phase 4 §3 "Mono usage" precedent for OID code-block + debug-stream event log).
- **Theme mode:** light + dark, dark default first paint, persisted to `localStorage['theme']`, toggle in AppShell.
- **Color tokens:** the existing `:root` and `.dark` CSS variable blocks (`app.css`) ARE the contract. Phase 6 references these tokens via Tailwind utilities (`bg-card`, `text-muted-foreground`, `border-destructive`, etc.). Phase 6 does NOT modify `app.css`. Phase 6 introduces ZERO new color tokens.
- **Motion:** shadcn defaults; Phase 6 adds three explicit motion contracts (cursor blink, dots pulse, chip expand) — all respect `prefers-reduced-motion`.
- **Accessibility floor:** WCAG 2.1 AA; `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2` on every interactive element; landmarks (`<nav aria-label="Primary">`, `<main>`) inherited from `AppShell`; Phase 6 adds the `<ol role="log" aria-live="polite">` transcript landmark.

### Phase 6 Typography Exceptions (documented; not part of the prescriptive scale)

- **Role label `text-[10px]`** (CONTEXT D-31 verbatim): 10px is BELOW the Phase 4 §3 Label size (12px). Justified: role labels in a flat thread are micro-affordances — at 12px they compete with body text and undermine the Linear-dense ethos. 10px with `uppercase tracking-wider` lifts them visually without weight inflation. This is a documented exception, used ONLY for `YOU` / `AGENT` role labels. NOT a new prescriptive token.
- **Mono fallback stack**: Phase 6 does not load a custom mono webfont. The `font-mono` Tailwind utility falls through to the system mono stack. Consistent with Phase 4 §3 OID code-block precedent. Mono is load-bearing for tool names, JSON args, and stringified tool outputs — three elements where misalignment would obscure agent reasoning. Plain text is rendered in Geist Variable per the rest of the design system.
- **Recharts label-on-bar utility class** (Phase 5 §7 SalaryBands): not relevant to Phase 6 — no charts in chat.

**Anti-choices reaffirmed**:

- No `text-base` (16px) — would break the 4-size cap. Body stays at `text-sm` (14px).
- No `font-medium` (500) in Phase 6's own component code (shadcn-internal usage like Button-sm label is unchanged from primitive defaults; counts under the inherited Phase 4 §3 contract).
- No italic in chat content. Tool-output JSON may render italic literals from the model's stringified output (rare); not a contract violation.

## 6. Component Specs

### 6a. ChatTranscript

**Location:** `frontend/src/components/chat/ChatTranscript.tsx`

**Purpose:** Scrollable `<ol role="log">` list of `TranscriptItem`s. Renders items in append order. Owns the smart-autoscroll behavior. Pure presentation — no fetch / no stream logic.

**Props (typed contract):**

```ts
type ChatTranscriptProps = {
  items: TranscriptItem[]
  isStreaming: boolean
  coldStart: boolean
  networkError: NetworkError | null
}
```

**Visual structure:**

```
┌─ <div className="flex-1 overflow-y-auto px-4 py-6"> ─────────────────────┐
│                                                                          │
│   ┌─ Network-error Alert (optional, top-of-list, mx-4 mt-4) ─┐           │
│   │ destructive Alert ── only when networkError !== null     │           │
│   └──────────────────────────────────────────────────────────┘           │
│                                                                          │
│   ┌─ <ol role="log" aria-live="polite" aria-relevant="additions"        │
│   │      className="space-y-4"> ──────────────────────────────┐         │
│   │                                                            │         │
│   │   <li>  ChatMessage variant="user-message" />              │         │
│   │   <li>  ChatMessage variant="assistant-text" streaming?/>  │         │
│   │   <li>  ToolChip (running|done) />                          │         │
│   │   <li>  ChatMessage variant="assistant-text" />            │         │
│   │   <li>  ChatMessage variant="error" />                     │         │
│   │   …                                                         │         │
│   │                                                            │         │
│   │   <div ref={bottomSentinelRef} aria-hidden="true" />       │         │
│   └────────────────────────────────────────────────────────────┘         │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

**Class strings (locked):**

| Selector | Class |
|----------|-------|
| Outer scroll container | `flex-1 overflow-y-auto px-4 py-6` |
| Network-error Alert wrapper (rendered conditionally) | `mx-4 mt-4` |
| `<ol>` element | `space-y-4` |
| `<li>` element | (no class — semantic only; ChatMessage / ToolChip control their own block layout) |
| Bottom sentinel `<div>` | `h-px w-px` (1×1 invisible; `aria-hidden="true"`; used for IntersectionObserver autoscroll detection) |

**Behavior:**

- Items render in array order. Each item gets `key={item.id}` (crypto.randomUUID assigned by `useChatStream` at append time; D-07).
- The empty-transcript case (zero items) is NOT handled here. The `Chat` route renders `<ChatEmptyState>` instead of `<ChatTranscript>` when `items.length === 0` (see §6e and §8).
- Smart autoscroll (see §8) is implemented via a `useEffect` on `items` length + `bottomSentinelRef` + IntersectionObserver. When the sentinel is `intersecting`, autoscroll on new items; when not (user has scrolled up), suppress.
- During streaming, the LAST item is typically an `assistant-text` with `streaming: true`. Its `<ChatMessage>` renders the blinking cursor (§6f).

**Accessibility:**

- `<ol role="log" aria-live="polite" aria-relevant="additions">` — screen readers announce new items as they append, without re-reading prior content.
- Per D-32: `aria-live="polite"` (NOT `assertive`) — assertive would interrupt the user's screen-reader navigation on every token. Polite is the WCAG-recommended choice for chat-style streams.
- `aria-relevant="additions"` — explicitly excludes `text` updates inside existing items (which would otherwise re-announce on every token append).

**Auto-grow behavior:**

The Textarea grows in height with content up to `max-h-[200px]` (~6 rows at 14px Geist), then begins scrolling. Implementation: a `useEffect` on `value` sets `textareaRef.current.style.height = 'auto'` then `= ${textareaRef.current.scrollHeight}px` (standard auto-grow pattern). **Keymap (D-15):**

| Key | Action |
|-----|--------|

| `Cmd+Enter` / `Ctrl+Enter` | NOT bound (D-15) |
| `Escape` | NOT bound on the Textarea in v1 (revisit if it conflicts with browser autocomplete behavior — Phase 8 polish window) |

**Disabled / state matrix (§14 below also documents):**

| State | Textarea | Send | Stop |
|-------|----------|------|------|
| Empty (no value, not streaming) | enabled, focused on mount | disabled (no value to send), accent fill | not rendered |
| Typing (value, not streaming) | enabled | enabled, accent fill | not rendered |
| Streaming | disabled (`disabled` prop) | not rendered | rendered, destructive fill, enabled |
| Network error (`disabled` prop true) | disabled | disabled | not rendered |

**JSX skeleton (locked structure):**

```tsx
import { useEffect, useRef } from 'react'
import { Send, Square } from 'lucide-react'
import { Button } from '@/components/ui/button'
import { Textarea } from '@/components/ui/textarea'

export function ChatComposer({
  value,
  onChange,
  onSubmit,
  onStop,
  isStreaming,
  disabled = false,
}: ChatComposerProps) {
  const textareaRef = useRef<HTMLTextAreaElement>(null)

  // Auto-grow textarea
  useEffect(() => {
    const ta = textareaRef.current
    if (!ta) return
    ta.style.height = 'auto'
    ta.style.height = `${ta.scrollHeight}px`
  }, [value])

  const handleKeyDown = (e: React.KeyboardEvent<HTMLTextAreaElement>) => {
    if (e.key === 'Enter' && !e.shiftKey) {
      e.preventDefault()
      if (!isStreaming && !disabled && value.trim() !== '') {
        onSubmit()
      }
    }
  }

  return (
    <div className="sticky bottom-0 border-t bg-background px-4 py-3">
      <div className="flex items-end gap-2">
        <Textarea
          ref={textareaRef}
          value={value}
          onChange={(e) => onChange(e.target.value)}
          onKeyDown={handleKeyDown}
          placeholder="Ask the agent something…"
          aria-label="Ask the agent"
          rows={1}
          className="min-h-[44px] max-h-[200px] resize-none flex-1"
          disabled={isStreaming || disabled}
        />
        {isStreaming ? (
          <Button
            type="button"
            size="sm"
            variant="destructive"
            onClick={onStop}
            aria-label="Stop streaming response"
          >
            <Square className="h-4 w-4 mr-1" aria-hidden="true" />
            Stop
          </Button>
        ) : (
          <Button
            type="button"
            size="sm"
            onClick={onSubmit}
            disabled={disabled || value.trim() === ''}
            aria-label="Send message"
          >
            <Send className="h-4 w-4 mr-1" aria-hidden="true" />
            Send
          </Button>
        )}
      </div>
    </div>
  )
}
```

4. **NO auto-submit.** D-25 verbatim — "pre-fill, no auto-submit". User reviews / edits / hits Enter.

**Accessibility:**

- Each chip uses `role="button"` + `tabIndex={0}` + `onKeyDown` for Enter/Space activation. (Badge is rendered as `<span>` under the hood by shadcn; not a native button.)
- `aria-label` is not added per chip because the chip's visible text IS the accessible name. Adding `aria-label` would either duplicate it (verbose) or override it (lying to screen readers).
- Focus-visible ring (`focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2`) inherited from shadcn Badge.

### 6f. Streaming Cursor

**Location:** `frontend/src/components/chat/ChatMessage.tsx` (inline component `<StreamingCursor>` exported nowhere else)

**Purpose:** Phase 4 §18 deferred item. Render a blinking caret `▍` (U+258D LEFT FIVE EIGHTHS BLOCK) at the end of a streaming assistant-text segment. Pure visual; `aria-hidden`. Renders flush against the last streamed character (no inline gap) — matches the Linear / Cursor convention and the CONTEXT.md D-21 specification verbatim.

**Visual structure:**

```tsx
<span className="animate-blink" aria-hidden="true">▍</span>
```

**Class strings (locked):**

| Property | Value | Rationale |
|----------|-------|-----------|
| Glyph | `▍` (U+258D) | Half-block height; reads as a software cursor without being noisy like full-block. Stripe Workbench / Linear use similar half-block glyphs. |
| Color | inherited from parent `<p>` (no class) → `--foreground` | Matches the body text it trails. |
| Animation | `animate-blink` — Tailwind custom utility (NOT shipped by default) | See §10 motion contract — Phase 6 adds the `@keyframes blink` to `app.css`. |
| `aria-hidden` | `true` | D-32 verbatim — cursor is decorative; screen readers must NOT announce it (would interrupt `aria-live` flow). |

**Animation timing (§17 discretion-resolved, default 1s per CONTEXT D-21):**

| Phase | Duration | Easing |
|-------|----------|--------|
| Visible | 500ms (half of 1s cycle) | `step-end` |
| Hidden | 500ms | `step-end` |

**Tailwind/CSS contract** (Phase 6 plan task adds this to `app.css`, NOT to `tailwind.config` — Tailwind v4 reads `@theme` / `@keyframes` from CSS):

```css
@keyframes blink {
  0%, 50% { opacity: 1; }
  50.01%, 100% { opacity: 0; }
}

.animate-blink {
  animation: blink 1s step-end infinite;
}

@media (prefers-reduced-motion: reduce) {
  .animate-blink {
    animation: none;
    opacity: 1; /* cursor stays visible, doesn't blink */

}
```

The `prefers-reduced-motion` carve-out per §10 keeps the visual affordance (you can see the cursor) without the motion. Matches Phase 4 §14 anti-motion rule.

**When the cursor renders:**

- When the current assistant-text item is `streaming: true` AND `content.length > 0` (i.e., at least one token has arrived; pre-first-token state is the "Thinking…" / "Warming up…" placeholder, NOT the cursor).
- Cursor disappears when:
  - `streaming: false` flips (item completes naturally on `final` event or on `tool_start` segmentation per D-08).
  - User clicks Stop (item is marked `stopped: true` AND `streaming: false`).
  - An ErrorEvent arrives mid-stream (the current text item closes; cursor disappears; error item appends).

| State | `isStreaming` | `coldStart` | Visual representation |
|-------|---------------|-------------|----------------------|
| `idle` | `false` | `false` | Composer ready; Send button if value, disabled-Send if empty; transcript shows finalized items or `<ChatEmptyState>` |
| `submitted` | `true` | `false` | Composer locked (Textarea disabled, Stop button shown); transcript shows user message just appended; no assistant item yet |
| `thinking` | `true` | `false` | Composer locked; transcript shows user message + assistant item with "Thinking…" + animated dots |
| `warming` | `true` | `true` | Composer locked; transcript shows user message + assistant item with "Warming up the agent — this can take ~4 minutes after idle" + animated dots |
| `streaming` | `true` | `false` (cleared on first event per D-19) | Composer locked; transcript shows partially-populated assistant text with blinking cursor; tool chips appear interleaved with their own running / done states |
| (transition: `final` event) | flips to `false` | `false` | Cursor disappears; assistant item content frozen; composer re-enables; Stop → Send |
| (transition: `error` event) | flips to `false` | `false` | Cursor disappears; current assistant item frozen at whatever content it had; error item appended below; composer re-enables; Stop → Send |
| (transition: user Stop) | flips to `false` | `false` | Cursor disappears; current assistant item gets `stopped: true` suffix "(stopped)"; composer re-enables; Stop → Send |

**HeartbeatEvent handling (D-20):** silently consumed — does NOT trigger a state transition. BUT the cold-start `setTimeout` IS cleared on receiving a heartbeat (since heartbeats only fire AFTER the backend's stream is alive); this is how the "Warming up" state self-corrects even before the first content token.

**Cold-start timer lifecycle (D-19):**

1. On `submit()`: clear any existing timer; set a new `setTimeout(10_000)` that flips `coldStart` to `true`.
2. On first non-heartbeat event AND on any heartbeat: clear the timer; set `coldStart` to `false`.
3. On stream end (final / error / stopped): clear the timer (defensive).

## 12. Cold-Start UX (Phase 4 §18 deferred item)

The literal item Phase 4 UI-SPEC §18 deferred: *"Cold-start 'warming' state distinct from 'thinking' state (Pitfall 4) — Phase 6 UI-SPEC will land this on top of the readSSEStream helper Phase 4 ships."*

Phase 6 lands it as follows:

**Timeline:**

```
0ms     ──┐ user submits
          ▼
        useChatStream.submit() runs:
          1. appends user-message TranscriptItem
          2. appends assistant-text TranscriptItem (content='', streaming=true)
          3. sets isStreaming=true
          4. sets coldStart=false
          5. starts setTimeout(10_000)
          6. starts authedFetch('/agent/stream', POST, signal)
          ▼
0–~225ms  ──┐ fetch resolves AT WARM (server-side latency is sub-second on warm container)
            ▼
~225ms      readSSEStream() begins yielding events
            first TokenEvent → clear timer, update content, render cursor → STATE: streaming
            (cold-start swap NEVER fires on warm path)

         OR cold path:
0–~225s   ──┐ fetch resolves AT COLD START (ACA scale-from-zero ~225s observed; this is the
            │  longest path documented in memory `aca-cold-start-profile.md`)
            ▼
10s         ──┐ setTimeout(10_000) fires
              ▼
              coldStart=true → assistant-text item swaps "Thinking…" → "Warming up the agent — this can take ~4 minutes after idle"
              (timer is now done; no further label changes)

~225s     ──┐ fetch resolves; first event arrives (or heartbeat arrives first)
            ▼
            clear cold-start state (coldStart=false) → render appropriate streaming UX
```

**Copy contract (verbatim, D-19):**

| Label | Copy |
|-------|------|
| Pre-first-event placeholder (0–10s) | `Thinking` + animated dots |
| Cold-start placeholder (10s+) | `Warming up the agent — this can take ~4 minutes after idle` + animated dots |

The em-dash (`—` U+2014, not a regular hyphen) is verbatim from the decision. Sentence-case per Phase 4 §13 (not "Warming Up The Agent…"). No emoji. No exclamation. Honest tone matches Phase 4 §13 voice rules.

**State management implementation contract** (executed in `useChatStream`):

```ts
function submit(query: string) {
  const userId = crypto.randomUUID()
  const assistantId = crypto.randomUUID()
  setItems((prev) => [
    ...prev,
    { kind: 'user-message', id: userId, content: query },
    { kind: 'assistant-text', id: assistantId, content: '', streaming: true },
  ])
  setIsStreaming(true)
  setColdStart(false)

  // Cold-start timer
  if (coldStartTimerRef.current) clearTimeout(coldStartTimerRef.current)
  coldStartTimerRef.current = setTimeout(() => setColdStart(true), 10_000)

  // Stream lifecycle (abridged — see useChatStream.ts spec in plan)
  const controller = new AbortController()
  abortControllerRef.current = controller
  void runStream(query, assistantId, controller.signal)
}

async function runStream(query: string, assistantId: string, signal: AbortSignal) {
  try {
    const response = await authedFetch('/agent/stream', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ query }),
      signal,
    })
    for await (const event of readSSEStream(response)) {
      // First event arriving clears cold-start
      if (coldStartTimerRef.current) {
        clearTimeout(coldStartTimerRef.current)
        coldStartTimerRef.current = null
      }
      setColdStart(false)
      // Dispatch event to reducer that mutates `items`...
    }
  } catch (err) {
    // see §9 network-error handling
  } finally {
    setIsStreaming(false)
    if (coldStartTimerRef.current) clearTimeout(coldStartTimerRef.current)
  }
}
```

(Full hook implementation is the planner / executor's responsibility; the contract above pins the cold-start semantics.)

**Why 10s threshold:**

- Warm ACA container on a typical query returns first event at <2s (verified by Phase 4 D-16 DebugAgentStream during smoke testing).
- 10s is well above warm-path latency (avoiding false-positive "warming up" labels on warm streams) and well below the observed cold-start (~225s; users see "warming up" for ~215s — long enough to set expectations).
- 5s would false-trigger on slow LLM first-token cases (some agent reasoning paths take 3–8s before first content token even on a warm container).
- 15s would leave users staring at "Thinking…" for too long on a cold start.
- 10s is the smallest value that's confidently AFTER warm and BEFORE cold-start frustration.

| `done-expanded` | `tool_end` received, user has expanded (`output !== null`, `expanded === true`) | `[ArrowRight] tool_name {compact_args_preview} [▼]` — chevron rotated 90° | `args` (pretty-printed JSON) + `output` (200-char preview) + optional "Show full output" Button |

**Transition cadence:**

- Running → done-collapsed: automatic on `tool_end` event; no user action.
- Done-collapsed → done-expanded: user click on header anywhere.
- Done-expanded → done-collapsed: user click on header (anywhere — not just chevron).
- Running → done-expanded: NOT directly possible (chip is disabled while running; user cannot expand until output arrives).

**FEATURES.md alignment:**

This chip pattern IS the "Inspectable agent tool calls" P1+VERY HIGH portfolio differentiator. The contract above is what enables the 30-second screen recording demo (ROADMAP Phase 6 success criterion 5): user submits → tool_start chip appears (running) → tool_end fills it (done-collapsed) → assistant streams synthesis around it → user clicks chip to expand and inspect → user clicks "Show full output" to inspect the raw JSON → final.

## 15. Density & Reading Order

Per D-31 verbatim: **density target = 4–6 messages visible in a 600px tall transcript viewport without scrolling.** Reads like Linear comments, not Discord.

**How the target is achieved:**

| Element | Vertical cost |
|---------|---------------|
| Per-message wrapper (`<li>`) | space-y-4 = 16px between items |
| Per-message internal | space-y-1 = 4px between role label and body |
| Role label | text-[10px] = ~14px line-height (10px × 1.4 leading-snug) |
| Body text (1 short line) | text-sm leading-normal = ~21px line-height (14px × 1.5) |
| **Total per short message** | 14 + 4 + 21 + 16 (gap below) = ~55px |
| **Six short messages in 600px** | 6 × 55 = 330px content + ~70px transcript padding = 400px → fits |
| **Realistic mixed transcript** | User-msg short + Assistant-text 2-line + Tool-chip collapsed + Assistant-text 1-line + User-msg short = ~310px → comfortably fits 5 turns in 600px |

**Visual reading order:**

1. Top: AppShell top-nav (h-12, inherited Phase 4 §7).
2. Below: transcript scroll container (`flex-1` consumes remaining viewport height).
3. Bottom: sticky composer (consumes its own height; transcript above shrinks accordingly).
4. Within transcript: items append at the BOTTOM (chronological); user reads top-to-bottom.
5. Smart-autoscroll keeps the latest content visible UNLESS user has scrolled up.
6. New items push older items up off-screen (when transcript exceeds viewport); user can scroll up to review.

**Chronological order of items within a single turn:**

```
user-message
  ↓ (16px gap)
assistant-text (may be empty + "Thinking…")
  ↓ (16px gap)
tool-call (running → done-collapsed)
  ↓ (16px gap)
assistant-text (synthesis around tool result)
  ↓ (16px gap)
tool-call (another)
  ↓ (16px gap)
assistant-text (final synthesis)
  [streaming ends on final event; no extra item]
  ↓ (16px gap)
[next user-message at start of next turn]
```

Per D-08: each turn's response is a LIST of items, not a single concatenated bubble. This makes the agent's reasoning legible without forcing the user to expand chips.

### Copywriting tone rules (Phase 6 affirmations of Phase 4 §13)

- **Voice:** Plain, direct, second-person. No marketing fluff. No emoji. (Inherited Phase 4 §13.)
- **Sentence case headings** (`Server restarting`, NOT `Server Restarting`). (Inherited.)
- **No oxford commas.** (Inherited.)
- **No exclamation marks** anywhere in Phase 6 copy. Honest tone for honest behaviors.
- **Em-dash U+2014, not double-hyphen `--`** — see `Warming up the agent — …` and `This shouldn't happen — …`. Linear / Stripe / Apple precedent.

### Phase 6 deliberate divergence from Phase 4 §13 "CTAs are verb + noun"

Phase 4 §13 closed: *"CTAs are verb + noun, never bare verbs: 'Copy ID' not 'Copy'; 'Reload page' not 'Reload'; 'Send query' not 'Send'."*

Phase 6 uses **bare verbs** on Send (`Send`) and Stop (`Stop`). Justification:

1. **Chat-input convention.** ChatGPT / Claude.app / Slack / Linear comments all label the primary chat-submit button with a bare verb or icon-only. "Send message" would feel verbose and ChatGPT-aspiring rather than Linear-dense.
2. **Context disambiguates noun.** The Textarea immediately above the Send button IS the noun. "Send" + an empty textarea is impossible (button disabled); "Send" + value is unambiguously "Send what's in the textarea". Phase 4's "Send query" was for the DebugAgentStream surface where the Input was labeled `Ask the agent something…` — the disambiguation was less visual.
3. **Icon supplies semantic enrichment.** `Send` button shows `<Send />` paper-plane glyph + label "Send". The icon resolves "send what?" → "send a message". `<Square />` + "Stop" resolves to "stop the streaming response" via context.
4. **aria-labels carry the full verb+noun.** `aria-label="Send message"` and `aria-label="Stop streaming response"` provide the full screen-reader semantic that Phase 4 §13 was protecting against context-loss for. The visible label can be bare.

**Phase 6 contract:** visible labels = bare verbs; aria-labels = verb+noun. This is the only documented exception to Phase 4 §13. All other CTAs (Show full output, Close, sample chips) follow Phase 4 §13 (or are full sentences in the case of sample chips).
