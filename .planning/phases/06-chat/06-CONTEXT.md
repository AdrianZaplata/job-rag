## Phase Boundary

Phase 6 ships the number-one portfolio-signal surface when the `/chat` route streams
tokens incrementally via `fetch`+`ReadableStream`, renders `tool_start`/`tool_end`
events as collapsed/expanded chips, marks completion on `final`, renders typed error
events in-transcript, and clears on refresh.

1. **Streaming token render** — `TokenEvent.content` appends to the current assistant
   text segment with smooth visual cadence. (CHAT-01, CHAT-02)
2. **Tool-call chips** — `ToolStartEvent` opens a collapsed chip showing
   `→ <tool_name>` + JSON args preview; the matching `ToolEndEvent` populates it with
   a 200-char output preview + "Show full" affordance. (CHAT-03, CHAT-04)
3. **Final event** — `FinalEvent` marks the assistant bubble complete and re-enables
   the composer; submitting during streaming is blocked at the UI layer. (CHAT-05)
4. **No history persistence** — refreshing the page clears the transcript entirely;
   no localStorage, no IndexedDB, no server-side session resume. (CHAT-06)
5. **Live demoability** — a screen recording of one chat turn (question → tool_start
   chip → tool_end chip → streamed synthesis → final) renders in <30 seconds against
   the deployed ACA stack and is portfolio-ready as-is. (ROADMAP §Phase 6 success
   criterion 5)

## Implementation Decisions

### A. Backend method change

- **D-01: `/agent/stream` switches from `GET ?q=...` to `POST {query: string}` body.**
  Three reasons:
  1. **Consistency with `/agent`** — the non-streaming sibling already accepts
     `AgentQuery(query: str)` via POST (`src/job_rag/api/routes.py:344-347`). Two
     entry points to the same agent with two different HTTP contracts is incidental
     complexity.
  2. **No URL-length ceiling** — chat queries may include pasted job descriptions,
     multi-line questions, or copy-pasted CV bullets. URL query params have a
     practical 2–4 KB ceiling across Envoy/CloudFront/browsers; request bodies
     don't.
  3. **Alignment with the Phase 4 D-16 debug probe assumption** — `DebugAgentStreamPage`
     was scaffolded to POST `{message: query}` (`frontend/src/routes/DebugAgentStream.tsx:35-39`).
     Phase 4 never live-exercised it against the actual backend (the page exists but
     a real send against the current `GET ?q=` backend would 405). Phase 6 closes the
     gap by aligning the backend to the assumed contract.

  Counterfactual considered: **keep GET, switch frontend to GET.** Lower-touch
  (2 lines of test changes vs 5), but the long-query and consistency-with-`/agent`
  arguments dominate. Rejected.

  Counterfactual considered: **support both GET and POST.** Doubles the OpenAPI
  surface, doubles the rate-limit policy, doubles the error path. Rejected — YAGNI.

- **D-02: Reuse the existing `AgentQuery` Pydantic model** for the POST body.
  `class AgentQuery(BaseModel): query: str` already lives at `routes.py:340-341`.
  Phase 6 changes the `/agent/stream` decorator from
  `@router.get("/agent/stream", ...)` to `@router.post("/agent/stream", ...)` and
  the handler signature from `async def agent_stream(request: Request, q: str)` to
  `async def agent_stream(request: Request, payload: AgentQuery)`. All other code
  inside the handler (timeout wrap, active_streams tracking, error frames, ping
  factory, shutdown_event, defensive headers) stays identical.

- **D-03: Test updates are mechanical** — three call sites in `tests/test_api.py`
  switch from `client.get("/agent/stream", params={"q": "..."})` to
  `client.post("/agent/stream", json={"query": "..."})` and one streaming-context
  call from `"GET", "/agent/stream?q=test"` to `"POST", "/agent/stream"` with a JSON
  body. No new test classes; this is a contract-rename refactor.

- **D-04: OpenAPI snapshot regen is required** —
  `frontend/openapi.snapshot.json` regenerates via the existing
  `npm run codegen:snapshot` script after the backend lands D-01/D-02. The
  drift-check CI job (Phase 4 D-14) will fail-loud if either side ships first; plans
  must sequence backend-then-frontend OR co-land in one PR.

- **D-05: Rate limit unchanged** — `agent_limit` (10/min, defined in
  `src/job_rag/api/auth.py`) stays attached. v1 single-user; the 10/min ceiling is
  generous for one human typing.

### B. Transcript & message model

- **D-06: Linear-dense flat thread, NOT ChatGPT-style alternating bubbles.**
  - User and assistant messages share the left column; a small role label
    (`text-xs uppercase tracking-wide text-muted-foreground`) marks each message
    ("You" / "Agent").
  - 16px (`gap-4`) between messages; no bubble border, no background tint, no
    avatar circles.
  - Reads like a Linear inbox or Stripe Workbench transcript rather than iMessage.
  - Matches PROJECT.md "Linear-style dense aesthetic — grayscale + one accent,
    small type, dense info" and Phase 4 D-20 (zinc + Geist + new-york + default
    dark). Counterfactual: bubble-style (ChatGPT) — rejected because it adds visual
    weight and competes with the tool-call chips for attention.

- **D-07: TranscriptItem discriminated union** as the in-memory shape:
  ```typescript
  type TranscriptItem =
    | { kind: 'user-message'; id: string; content: string }
    | { kind: 'assistant-text'; id: string; content: string; streaming: boolean }
    | { kind: 'tool-call'; id: string; name: string; args: object | null;
        output: string | null; expanded: boolean }
    | { kind: 'error'; id: string; reason: ErrorEvent['reason']; message: string }
  ```
  Each `TranscriptItem` has a stable `id` (crypto.randomUUID at append) so React
  reconciles correctly. Tool-call items hold both start (args) and end (output)
  state; `output === null` means the tool is still running, render the spinner.

- **D-08: Interleaved tool calls with text segments.** When a `ToolStartEvent`
  arrives mid-stream, the current `assistant-text` item closes (`streaming: false`),
  a new `tool-call` item appends, and subsequent `TokenEvent`s open a fresh
  `assistant-text` item. Each turn's response is a list of items:
  `[assistant-text, tool-call, assistant-text, tool-call, assistant-text]` — not
  one big bubble. This makes the agent's reasoning legible without forcing the user
  to expand chips to follow along.

- **D-09: Within-session history is append-only.** Each user submit appends a new
  `user-message` item; agent response items append after. Refresh wipes the React
  state — no localStorage, no IndexedDB. The backend agent itself is single-turn
  (each `/agent/stream` call is independent — no conversation context passed
  forward). The visible history is purely a UI affordance to review prior turns
  within the same session. **Per CHAT-06**, refresh clears entirely with zero
  storage residue.

### C. Tool-call chip UX (THE differentiator per FEATURES.md)

- **D-10: Collapsed by default, click anywhere on the chip header to expand.**
  Layout (collapsed):
  ```
  [arrow icon] → search_jobs  {"query": "Azure senior Berlin"}  [chevron]
  ```
  - `[arrow]` icon: `ArrowRight` from lucide-react (already in deps via shadcn)
  - Tool name: `font-mono text-sm`
  - JSON args preview: `font-mono text-xs text-muted-foreground`, single-line
    truncated with `truncate` at the container width
  - `[chevron]`: `ChevronRight` rotates 90° (`rotate-90` class) when expanded

  Layout (expanded):
  ```
  [arrow] → search_jobs
    args:
      {
        "query": "Azure senior Berlin",
        "seniority": "senior"
      }
    output:
      [first 200 chars of stringified output]
      [Show full output] ← shadcn Button, link variant
  ```
  - shadcn `collapsible` primitive (NEW — install via `npx shadcn@latest add collapsible`)
  - args block: `<pre className="text-xs font-mono whitespace-pre-wrap">`
    rendering `JSON.stringify(args, null, 2)`
  - output preview: literal `.slice(0, 200)` + `"…"` if longer, in a `<pre>` block
  - "Show full output" button reveals full output in a shadcn `Dialog` (Phase 4
    primitive) — keeps the transcript inline-readable but allows full inspection on
    demand

- **D-11: Args formatting = pretty-printed JSON.** `JSON.stringify(args, null, 2)`
  for the expanded view. For the collapsed preview, render as
  `JSON.stringify(args)` (compact) and truncate visually with `truncate`. The
  literal CHAT-03 wording is "JSON-preview of args" — pretty-print on expand
  honors the spirit (legibility) while compact-collapse honors the constraint
  (single-line chip).

- **D-12: Output truncation per CHAT-04 = 200 chars + "expand" affordance.**
  - Truncated string: `output.slice(0, 200)` + `"…"` (literal ellipsis char)
  - "expand" affordance: shadcn Button (link variant) labeled "Show full output"
    that opens a Dialog with the full string in a scrollable `<pre>` block
  - Output is already stringified server-side per sse.py `ToolEndEvent.output: str`
    field (Phase 1 D-04); the frontend never needs to JSON-decode the output

- **D-13: While tool is running (`tool_start` received, `tool_end` not yet) the
  chip shows a small spinner next to the tool name.** A pulsing dot
  (`<span className="animate-pulse">·</span>`) is enough; full Skeleton is
  overkill for ~200ms tool calls. Closes on `tool_end` — spinner replaced by the
  output preview, expand affordance becomes available.

### D. Composer (input) UX

- **D-14: Pinned-to-bottom multi-line composer.**
  - `<Textarea>` (shadcn primitive — install in this phase via
    `npx shadcn@latest add textarea`) with `rows={1}` and `min-h-[44px]` +
    `max-h-[200px]` + `overflow-y-auto` so it grows up to ~6 rows then scrolls.
  - Container: `sticky bottom-0` inside the route scrolled-element with
    `bg-background border-t pt-3 pb-4 px-4`.
  - Send button: shadcn Button (default variant) labeled "Send" with paper-plane
    icon (`Send` from lucide-react). Disabled when textarea is empty or while
    streaming.
  - Counterfactual: in-flow composer at end of transcript (Stripe-style) — rejected
    because long transcripts force scrolling back to the bottom to type; sticky is
    universally expected for chat UIs.

- **D-15: Enter submits, Shift+Enter inserts newline.**
  - Default keymap matches ChatGPT / Claude.app / Linear comments.
  - `onKeyDown={(e) => if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault();
    submit(); }}`.
  - Cmd+Enter (Mac) / Ctrl+Enter (PC) — not bound; Enter is enough. Sample chips
    do not auto-submit; they pre-fill and let the user edit before sending.

- **D-16: Stop button replaces Send during streaming.**
  - During streaming: Send button hides; Stop button (red destructive variant)
    appears in the same slot with `<Square>` icon + "Stop" label.
  - Stop click: `abortController.abort()` → fetch terminates → `readSSEStream`
    throws AbortError → useChatStream catches → marks current `assistant-text`
    item with a `(stopped)` muted-foreground suffix → re-enables composer.
  - AbortController is freshly created on each submit; stored in a ref so Stop
    can reach it. New submit while streaming is prevented by the disabled
    composer; defensive auto-abort guards against any race.

- **D-17: Smart autoscroll to bottom.**
  - Default: autoscroll to bottom on every new token / new item.
  - Exception: if user has scrolled up (detected via `IntersectionObserver` on a
    bottom sentinel div, or simpler: scrollHeight - scrollTop - clientHeight > 100),
    suppress autoscroll until they scroll back down OR submit a new query.
  - Linear-style: respects the user's intent to look back at earlier output without
    yanking the viewport.

### E. Streaming UX states

- **D-18: Pre-first-token placeholder = "Thinking…" with animated dots.**
  - After submit, immediately append an `assistant-text` item with
    `content: ''` and `streaming: true`. Render it with a `Thinking…` label
    + three CSS-animated dots (`<span className="animate-pulse">.</span>` × 3
    with staggered delays).
  - First `TokenEvent` flips: content starts populating, dots disappear, the
    label clears. Smooth — no layout shift.

- **D-19: Cold-start awareness — distinct "Warming up…" state.**
  - If no event has arrived 10s after the fetch resolved (which itself can take up
    to 225s on full cold start per memory `aca-cold-start-profile.md`), swap the
    "Thinking…" label for "Warming up the agent — this can take ~4 minutes after
    idle".
  - At 30s, no further change (the message already explains the wait).
  - Timer cleared on first non-heartbeat event.
  - Tracked via a `setTimeout(10_000)` on stream start; cleared in the event loop's
    first iteration that receives a real event.
  - This is the literal Phase 4 UI-SPEC §18 deferred item ("Cold-start 'warming'
    state distinct from 'thinking' state (Pitfall 4)") Phase 6 owns.

- **D-20: HeartbeatEvents are silently consumed.** They reset a "stream is alive"
  internal timer (used for cold-start state above). No UI surface; chat traffic
  shouldn't be polluted with infrastructure pings. The heartbeat IS the proof of
  life — its absence is what would trigger a "connection lost" state, but the
  fetch + ReadableStream already throws on connection drop, so we route through
  the error branch instead.

- **D-21: Streaming cursor = blinking caret at the end of the current
  assistant-text item.**
  - `<span className="animate-blink">▍</span>` after the streaming content;
    disappears when the text segment closes (`streaming: false` flips on
    `tool_start` or `final`).
  - Animation: 1s steady-state, 1s blink, repeat — slower than terminal cursors
    for readability.

### F. Error rendering

- **D-22: Each `ErrorEvent` renders an inline destructive `Alert` block at the
  message position where streaming stopped.**
  - shadcn `alert` primitive (already in deps via Phase 5)
  - Variant: `destructive`
  - Title: per-reason copy (see D-23)
  - Body: sanitized backend message + optional retry hint
  - Composer re-enables immediately after error; user can re-ask.
  - No retry button in v1 — user re-submits manually. Counterfactual considered:
    "Retry" button that re-sends the last user message. Rejected for v1 simplicity
    (one less state-tracking concern); revisit if Adrian sees friction.

- **D-23: Per-reason copy (closed Literal set from `ErrorEvent.reason`):**
  - `agent_timeout` → Title: **"Agent timed out"**, Body: "The agent took longer
    than 60 seconds to respond. Try a more specific question, or check the agent
    logs."
  - `shutdown` → Title: **"Server restarting"**, Body: "Container is shutting
    down. Please try again in a moment."
  - `llm_error` → Title: **"Language model error"**, Body: "The LLM call failed.
    Please try again."
    (Note: backend doesn't currently emit `llm_error` — the Literal includes it
    for future-proofing per Phase 1 D-19. Copy lands as a safety net.)
  - `internal` → Title: **"Something went wrong"**, Body: the sanitized backend
    message verbatim (already 200-char-truncated per Phase 1 `_sanitize`).

- **D-24: Network errors (fetch throws before any event)** = generic destructive
  Alert outside the transcript flow:
  - 401 → "Session expired. Please refresh." (authedFetch should have already
    redirected via Phase 4 D-11 InteractionRequiredAuthError path, so this is a
    backstop)
  - 403 → "Not authorized. (This shouldn't happen — file an issue if you see this.)"
  - 5xx → "Server error. Please try again."
  - Network unreachable → "Can't reach the server. Check your connection."

### H. Cancellation, abort, and resource cleanup

- **D-26: Single in-flight stream at a time.**
  - Composer is disabled while a stream is in flight, so the user can't submit a
    second query mid-stream.
  - The AbortController for the in-flight stream lives in a ref. If component
    unmounts (e.g., navigation), `useEffect` cleanup calls `abort()` to terminate
    the fetch and release the reader. Resource leak guard.
  - If Phase 8 adds keyboard shortcuts that allow rapid-fire submits, the
    auto-abort-previous mechanism is ready; no rework needed.

### K. Linear-dense aesthetic for chat

### L. Accessibility

- **D-32: Inherit Phase 4 UI-SPEC §15 a11y patterns + chat-specific additions:**
  - Transcript: `<ol role="log" aria-live="polite" aria-relevant="additions">`
    so screen readers announce new agent tokens without re-reading prior content.
  - Tool chips: `<button aria-expanded={expanded}>` for the header,
    `aria-controls={contentId}` linking to the expanded body.
  - Composer Textarea: `aria-label="Ask the agent"`; placeholder text duplicates

  - Streaming cursor: pure visual; not announced by screen readers (`aria-hidden`).
  - Stop button: explicit `aria-label="Stop streaming response"`.

### Claude's Discretion

- **Exact lucide-react icon choices** for the chips (`ArrowRight` vs `Zap` vs
  `Wrench`) — `ArrowRight` recommended for "call" semantics but the planner can
  swap if it conflicts with the AppShell nav iconography.
- **Animation timing for the blinking cursor** (0.8s vs 1s vs 1.2s) — taste call;
  default to 1s.
- **Exact pixel widths for the chip expanded args/output `<pre>` blocks** — let
  the container width drive; clamp to `max-w-full` so chips never overflow the
  transcript column.
- **`scrollIntoView` vs `scrollTop = scrollHeight` for autoscroll** — both work;
  `scrollIntoView({ behavior: 'smooth', block: 'end' })` reads cleaner; planner
  to verify smooth performance on rapid token streams (≥100 tokens/s) and fall
  back to instant scroll if jank.
- **Whether to render heartbeat events in DEV builds** — Debug page (Phase 4
  D-16) shows them; production Chat does not. Keep that asymmetry.
- **Sample query phrasing** in D-25 — exact strings are placeholders; planner /
  UI-SPEC researcher may refine to match the actual corpus shape (e.g., if the
  corpus has 0 staff-level postings, swap "staff" for "lead").
- **Whether ChatComposer's Stop button uses `Square` (filled) or `XCircle`
  (close) icon** — `Square` reads more clearly as "stop streaming"; defer to
  UI-SPEC review if not.
- **Markdown rendering in assistant text** — defer. v1 renders raw text. If the
  agent's responses include markdown (it does — backticks, lists), Phase 8
  portfolio polish can add a markdown renderer (`react-markdown` ~10 KB
  gzipped). Plain text first, see how it reads on a real demo.
- **Code-block syntax highlighting** in tool outputs — defer. Tool outputs are
  rarely code; if they are, the Dialog's `<pre>` block is enough.
- **Phase 6 commit grain** — Phase 5 used 6 plans (Wave 0 foundation → backend
  service → backend routes → frontend data layer → frontend widgets → UAT
  runbook). Phase 6 likely needs fewer plans (~4): Wave 0 foundation + backend
  method change + frontend chat surface + UAT runbook. Planner to decide.
